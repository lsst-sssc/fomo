## Integration Findings

### BLOCKER: `_coarse_telescope_label` fallback is broken for SOAR records

**File:** `solsys_code/management/commands/sync_lco_observation_calendar.py:274-298`, consumed at line 408.

```python
if len(instrument_type) >= 3:
    candidate = instrument_type[:3].lower()
    if candidate in {'0m4', '1m0', '2m0', '4m0'}:
        return candidate
return instrument_type
```

Phase 6's `_extract_instrument()` correctly returns SOAR's science-config instrument type (e.g. `SOAR_GHTS_REDCAM`), but Phase 7's coarse-fallback function assumes every instrument_type string is LCO-style aperture-class-prefixed (`'1M0-SCICAM-SINISTRO'`). `'SOAR_GHTS_REDCAM'[:3].lower()` is `'soa'`, not in the recognized set, so the function falls through to returning the full raw instrument string. Combined with `_title_for`'s `f'[UNVERIFIED] {telescope} {instrument}'` template, a SOAR record whose live API call fails would produce the title `[UNVERIFIED] SOAR_GHTS_REDCAM SOAR_GHTS_REDCAM` — not a coarse aperture-class label.

`SITE_TELESCOPE_MAP`'s `('sor', '4m0')` entry is correct and unaffected; this only breaks the *fallback* path (used when the live API call fails/times out/returns an unmapped code), not the verified-success path.

### BLOCKER (coverage gap): No test ever drives a SOAR record through the placed/live-API path

Every SOAR fixture in `test_sync_lco_observation_calendar.py` (`test_select_03/04/05`, `test_extract_02_soar_multi_config_*`) is banner-stage only — none sets `scheduled_start`. None of Phase 7's tests (`test_telescope_02/03/04`, `test_sync_06/07/09`, `test_d01_*`) use `facility='SOAR'`. The demo notebook's Phase 7 cells are LCO-only too. This is why the defect shipped with 35/35 green tests, a passed Nyquist validation, a passed security audit, and a passed UAT — none of those exercised the one combination (SOAR + placed + API-failure) that breaks.
