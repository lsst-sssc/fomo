## start-time-idempotency-key — load_telescope_runs silently duplicated CalendarEvents on re-ingest because its lookup key required exact equality on a drift-prone computed sunset time
- **Date:** 2026-07-08
- **Error patterns:** load_telescope_runs, insert_or_create_calendar_event, get_or_create, CalendarEvent, duplicate, idempotency, start_time, sun_event, IERS, Earth-orientation, UT1-UTC, AltAz, astropy, exact-match lookup key, telescope_runs, SYNC-04, silent duplicate creation
- **Root cause:** The create-or-update lookup key {telescope, instrument, start_time} used exact datetime equality on start_time, a value computed fresh each run by telescope_runs.sun_event(). sun_event() is deterministic within a process but its output is a direct function of astropy's IERS Earth-orientation data (UT1-UTC / polar motion) applied in the get_sun/AltAz transform, which astropy refreshes over time — so independent ingests of the identical (site, night) days apart produced start_time values ~2s apart, defeating get_or_create's exact match and creating near-duplicate rows.
- **Fix:** Added an optional `start_time_tolerance` kwarg to the shared insert_or_create_calendar_event() helper; when set (only load_telescope_runs passes it, at 5 minutes) it matches an existing event whose start_time is within a +/- window (start_time__range, a proximity window rather than minute-bucketing which would still split events across a bucket boundary) scoped by the other lookup keys, leaving the stored start_time pinned to avoid churn. Default None keeps exact get_or_create behaviour, so URL-keyed sync callers (sync_lco/sync_gemini, lookup {'url': ...}) are unaffected.
- **Files changed:** solsys_code/calendar_utils.py, solsys_code/management/commands/load_telescope_runs.py, solsys_code/tests/test_calendar_utils.py, solsys_code/tests/test_load_telescope_runs.py, docs/notebooks/pre_executed/load_telescope_runs_demo.ipynb

## site-search-mpc-no-match — site-search widget returned "No matches" for real MPC codes (G96/Mt. Lemmon) because list-shaped old_names silently discarded the entire MPC candidate pool
- **Date:** 2026-07-16
- **Error patterns:** site-search, site_selection, build_site_candidates, _flatten_mpc_candidates, MPCObscodeFetcher, query_all, old_names, list, TypeError, unhashable type: 'list', dict membership, candidate pool, No matches for this search, mpc_obscode_candidates, FileBasedCache, cache poisoning, degraded fallback, TTL, approval queue, Sites Needing Review, CampaignRun, G96, Mt. Lemmon Survey
- **Root cause:** _flatten_mpc_candidates() treated each MPC record's `old_names` as a string, but the live MPC bulk obscodes API (MPCObscodeFetcher.query_all()) returns it as a JSON **list** for 60 of 2712 records (e.g. G96 -> ['Mt. Lemmon Survey']; 0 are strings). The dict-membership test `if candidate and candidate not in mapping:` used the list as an operand, raising `TypeError: unhashable type: 'list'`. build_site_candidates()'s broad `except (RequestException, ValueError, KeyError, TypeError, AttributeError)` swallowed it and fell back to a local-only 23-string pool, discarding all 2712 MPC candidates — then cached that degraded pool under 'mpc_obscode_candidates' for 24h (FileBasedCache shared with runserver), so any MPC-only obscode never surfaced in the widget.
- **Fix:** New _old_name_strings() helper normalizes old_names (list | str | None/missing) into a flat list of non-empty strings; _flatten_mpc_candidates() folds each prior name in as its own candidate, so a list-valued old_names no longer reaches a dict key/membership op. Degraded (MPC-failed) fallback pool now cached for only MPC_CANDIDATE_FALLBACK_TTL_SECONDS (60s) instead of 24h, so a transient MPC outage can't poison search for a day; full pool still cached 24h. 4 regression tests added. RELATED LATENT BUG (same root-cause family, separate session site-resolve-list-old-names): to_observatory() assigns list-shaped old_names to Observatory.old_names (a CharField) on the tier-2 resolve-and-create path.
- **Files changed:** solsys_code/campaign_utils.py, solsys_code/tests/test_campaign_approval.py

## site-resolve-list-old-names — placeholder-replacement "Could not resolve that site" because the widget's combined "display (obscode)" click value never round-tripped through the candidate pool
- **Date:** 2026-07-16
- **Error patterns:** Could not resolve that site, site_selection, _resolve_site, resolve_site, build_site_candidates, selection_to_obscode, oversized obscode, _MAX_OBSCODE_LEN, site_search_results.html, onclick, display (obscode), placeholder Observatory, NEEDS REVIEW, approval queue, Resolve action, approve action
- **Root cause:** The site-search suggestion fragment's onclick handler writes the COMBINED string `f'{display} ({obscode})'` (e.g. 'Lowell Discovery Telescope (G37)') into the `site_selection` input on click. Both `CampaignRunDecisionView._resolve_site()` and its approve-path sibling mapped that selection back to an obscode via `build_site_candidates().get(selection, selection)`, but the pool is keyed only on bare display strings and bare obscodes — never the combined form. The exact lookup missed, the 32-char combined string passed through unchanged, and `resolve_site()` rejected it as an oversized obscode (`len > _MAX_OBSCODE_LEN=4`), returning `(None, True)` — surfaced as the generic "Could not resolve that site" error for a genuine MPC candidate the user had just selected from the list. Initially misdiagnosed as the same `old_names`-list issue as the G96 bug above; disconfirmed — the single-code `query()` endpoint returns `old_names=None` for G37, not a list, and `to_observatory()` succeeds fine.
- **Fix:** New `selection_to_obscode()` helper in `campaign_utils.py`: exact pool hit -> else parse the widget's trailing `' (obscode)'` token (preferring a pool hit on the leading display part) -> else passthrough. Both call sites (`_resolve_site()` and the approve path) now use it instead of the raw `build_site_candidates().get(selection, selection)`. Backward-compatible with bare obscodes/display strings.
- **Files changed:** solsys_code/campaign_utils.py, solsys_code/campaign_views.py, solsys_code/tests/test_campaign_approval.py

## site-search-degraded-pool-recurrence — site-search "No matches" regression caused by the test suite and the dev runserver sharing one FileBasedCache key
- **Date:** 2026-07-16
- **Error patterns:** No matches for this search, mpc_obscode_candidates, FileBasedCache, shared cache, cache.clear(), test isolation, override_settings, LocMemCache, cross-contamination, warmed pool wiped, cold rebuild, degraded fallback, name_utf8, short_name
- **Root cause:** `settings.CACHES` is a `FileBasedCache` under `/tmp`; Django does not isolate that cache between the test runner process and the dev `runserver` process — both read/write the same `mpc_obscode_candidates` key. Running the site-resolve-list-old-names bug's verification suite (which calls `cache.clear()` in setup) immediately before a browser test wiped the runserver's warmed ~5,700-entry MPC pool, forcing a cold rebuild that briefly served the tiny local-only fallback pool (no G37) — reproducing the exact "No matches" symptom the G96 fix had already resolved, for terms that had worked minutes earlier. The "another live-MPC record shape throws" hypothesis (same class as the G96 bug) was checked and disconfirmed — `build_site_candidates()` rebuilds a clean full pool from live data with no exception.
- **Fix:** Cache-touching test classes in `test_campaign_approval.py` and `test_campaign_site_search.py` now use `@override_settings(CACHES=<LocMemCache>)` so running tests can never again evict the shared runserver cache. Also hardened `_flatten_mpc_candidates()` defensively (new `_candidate_str()` helper) so `name_utf8`/`short_name` degrade per-field instead of dropping the whole pool if either ever comes back non-string-shaped in the future (same failure family as `old_names`, not yet observed for these two fields in live data).
- **Files changed:** solsys_code/campaign_utils.py, solsys_code/tests/test_campaign_approval.py, solsys_code/tests/test_campaign_site_search.py

## duplicate-mpc-candidate-match — site-search suggestion dropdown showed the same site twice because whitespace-variant candidate strings survived exact-string dedup
- **Date:** 2026-07-16
- **Error patterns:** duplicate suggestion, duplicate candidate, site-search, dropdown, Nordic Optical Telescope, name_utf8, old_names, whitespace, candidate not in mapping, exact-string dedup, visual duplicate, build_site_candidates, _flatten_mpc_candidates, _local_observatory_candidates
- **Root cause:** A single MPC record can carry two byte-distinct candidate strings that render identically once whitespace collapses in HTML — e.g. Z23's `name_utf8` is `'Nordic Optical Telescope, La Palma'` (1 space) but its `old_names` entry is the same text with 5 spaces after the comma. The existing pool-build dedup (`if candidate and candidate not in mapping`) is exact-byte, so both survive as separate keys, both substring-match the same query, and the suggestion dropdown shows the identical text twice. Live audit found 32 of 2,712 MPC records affected (mostly trailing-space `old_names` variants) — systemic, not Z23-specific.
- **Fix:** New `_normalize_candidate()` helper (`' '.join(value.split())` — collapses whitespace runs, matching how HTML renders text) applied to each candidate string BEFORE the first-seen dedup check, in both `_flatten_mpc_candidates()` (MPC pool) and `_local_observatory_candidates()` (local pool), so visual-duplicate keys collapse to one at the pool's source rather than needing per-consumer fixes.
- **Files changed:** solsys_code/campaign_utils.py, solsys_code/tests/test_campaign_site_search.py
