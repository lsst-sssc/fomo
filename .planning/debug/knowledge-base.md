## start-time-idempotency-key — load_telescope_runs silently duplicated CalendarEvents on re-ingest because its lookup key required exact equality on a drift-prone computed sunset time
- **Date:** 2026-07-08
- **Error patterns:** load_telescope_runs, insert_or_create_calendar_event, get_or_create, CalendarEvent, duplicate, idempotency, start_time, sun_event, IERS, Earth-orientation, UT1-UTC, AltAz, astropy, exact-match lookup key, telescope_runs, SYNC-04, silent duplicate creation
- **Root cause:** The create-or-update lookup key {telescope, instrument, start_time} used exact datetime equality on start_time, a value computed fresh each run by telescope_runs.sun_event(). sun_event() is deterministic within a process but its output is a direct function of astropy's IERS Earth-orientation data (UT1-UTC / polar motion) applied in the get_sun/AltAz transform, which astropy refreshes over time — so independent ingests of the identical (site, night) days apart produced start_time values ~2s apart, defeating get_or_create's exact match and creating near-duplicate rows.
- **Fix:** Added an optional `start_time_tolerance` kwarg to the shared insert_or_create_calendar_event() helper; when set (only load_telescope_runs passes it, at 5 minutes) it matches an existing event whose start_time is within a +/- window (start_time__range, a proximity window rather than minute-bucketing which would still split events across a bucket boundary) scoped by the other lookup keys, leaving the stored start_time pinned to avoid churn. Default None keeps exact get_or_create behaviour, so URL-keyed sync callers (sync_lco/sync_gemini, lookup {'url': ...}) are unaffected.
- **Files changed:** solsys_code/calendar_utils.py, solsys_code/management/commands/load_telescope_runs.py, solsys_code/tests/test_calendar_utils.py, solsys_code/tests/test_load_telescope_runs.py, docs/notebooks/pre_executed/load_telescope_runs_demo.ipynb

## site-search-mpc-no-match — site-search widget returned "No matches" for real MPC codes (G96/Mt. Lemmon) because list-shaped old_names silently discarded the entire MPC candidate pool
- **Date:** 2026-07-16
- **Error patterns:** site-search, site_selection, build_site_candidates, _flatten_mpc_candidates, MPCObscodeFetcher, query_all, old_names, list, TypeError, unhashable type: 'list', dict membership, candidate pool, No matches for this search, mpc_obscode_candidates, FileBasedCache, cache poisoning, degraded fallback, TTL, approval queue, Sites Needing Review, CampaignRun, G96, Mt. Lemmon Survey
- **Root cause:** _flatten_mpc_candidates() treated each MPC record's `old_names` as a string, but the live MPC bulk obscodes API (MPCObscodeFetcher.query_all()) returns it as a JSON **list** for 60 of 2712 records (e.g. G96 -> ['Mt. Lemmon Survey']; 0 are strings). The dict-membership test `if candidate and candidate not in mapping:` used the list as an operand, raising `TypeError: unhashable type: 'list'`. build_site_candidates()'s broad `except (RequestException, ValueError, KeyError, TypeError, AttributeError)` swallowed it and fell back to a local-only 23-string pool, discarding all 2712 MPC candidates — then cached that degraded pool under 'mpc_obscode_candidates' for 24h (FileBasedCache shared with runserver), so any MPC-only obscode never surfaced in the widget.
- **Fix:** New _old_name_strings() helper normalizes old_names (list | str | None/missing) into a flat list of non-empty strings; _flatten_mpc_candidates() folds each prior name in as its own candidate, so a list-valued old_names no longer reaches a dict key/membership op. Degraded (MPC-failed) fallback pool now cached for only MPC_CANDIDATE_FALLBACK_TTL_SECONDS (60s) instead of 24h, so a transient MPC outage can't poison search for a day; full pool still cached 24h. 4 regression tests added. RELATED LATENT BUG (same root-cause family, separate session site-resolve-list-old-names): to_observatory() assigns list-shaped old_names to Observatory.old_names (a CharField) on the tier-2 resolve-and-create path.
- **Files changed:** solsys_code/campaign_utils.py, solsys_code/tests/test_campaign_approval.py
