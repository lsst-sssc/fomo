<objective>
Wire the Plan 01 live-search endpoint into the two site-entry surfaces: the public submission
form's `site_raw` field (D-09 — no create-new link for public submitters) and the approval-queue
pending row's inline site input (D-10 — replace the static ≤5-candidate `<datalist>` with the live
widget, keeping the "Create new Observatory" staff escape hatch). Both are text-only fills: picking
a suggestion populates the input; site resolution still happens at approval (D-06), so no model
change, no migration, and no DB write is triggered by anonymous typing.

Purpose: A submitter/staff member can now search by typing text *different* from the original
`site_raw` and get live substring-first suggestions (correcting a submitter's typo, finding a site
by partial name), instead of Phase 21's one-shot datalist computed against the originally-submitted
text.

Output: `CampaignRunSubmissionForm.site_raw` gains `hx-*` widget attrs + a crispy suggestions
container; `ApprovalQueueTable.render_site()`'s pending-row branch swaps its `<datalist>` for the
live HTMX widget; markup regression tests on both surfaces.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| submitter/staff keystrokes → SiteSearchView (via widget hx-get) | Widget-originated queries reach the Plan 01 endpoint; the widgets themselves add no new server trust boundary (fill is text-only, resolution deferred). |
| picked suggestion text → site_raw / site_selection input value | Untrusted picked string becomes the input value; it is never resolved to an Observatory until a staff approve/resolve action (D-06). |

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |

| T-22-04 | Tampering | render_site() pending-row markup | high | mitigate | All interpolated values (`site_raw`, pk, urls) go through `format_html`/`format_html_join` positional auto-escaping (T-21-01); no f-string/mark_safe. The suggestion fragment itself (endpoint side) is escaped in Plan 01. |
| T-22-05 | Elevation of Privilege | picked suggestion fills input text only (D-06) | low | accept | The widget performs zero DB writes and no resolution; anonymous traffic can never trigger `resolve_site` tier-2 Observatory creation (that only fires from a staff approve/resolve action). Text-only fill is by design. |
