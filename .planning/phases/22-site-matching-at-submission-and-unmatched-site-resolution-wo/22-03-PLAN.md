<objective>
Close the last Phase 21 gap: approved runs whose site never resolved (`site_needs_review=True`)
are currently a dead end — no CalendarEvent, no UI to fix them. This plan adds (1) a third
"Sites Needing Review" table on the existing approval-queue page listing those runs (D-07,
reusing `ApprovalQueueTable` + the page's once-per-request candidate pool), each row carrying
the Plan 02 live-search widget plus a single "Resolve" action; and (2) a new `resolve_site`
action on `CampaignRunDecisionView.post()` (D-08) that resolves the site via the existing
display-string→obscode pool mapping + `resolve_site(create_placeholder=False)`, honoring the
Phase 21 D-06 never-re-resolve guard, then fires the deferred CalendarEvent projection in the
same request. The projection block currently inlined in the approve branch is factored into a
shared `_project_calendar_event(run)` helper called by both approve (inside its UNCHANGED
revert-on-failure try/except) and resolve_site (inside its own NON-reverting try/except).

Purpose: "Site failure never blocks approval" stays true, but an approved run with an
unresolved site now has a resolution workflow instead of requiring a manual DB edit — and
resolving it retroactively projects the calendar event approval skipped.

Output: `_project_calendar_event()` helper; a `resolve_site` decision-view branch; a third
table in `ApprovalQueueView`; resolve-mode rendering in `ApprovalQueueTable`; the
`approval_queue.html` third-table block; and a `TestSitesNeedingReview` test class.
