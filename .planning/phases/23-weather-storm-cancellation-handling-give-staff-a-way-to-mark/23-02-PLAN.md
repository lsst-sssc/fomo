<objective>
Give staff a non-admin, approval-queue way to mark an already-approved `CampaignRun` as
`CANCELLED` or `WEATHER_TECH_FAILURE`, and keep the linked calendar event in sync (D-03, D-04,
D-05). Today `run_status` is only editable in Django admin; the Decided table renders no actions at
all. This plan adds the first Decided-row action, two distinct terminal title prefixes on the
linked `CAMPAIGN:{pk}` `CalendarEvent`, and the `[WEATHERED]` box-shadow ring.

Purpose: When a storm wipes an approved campaign run, staff click a button on the Decided table and
the run is marked weathered/cancelled with the loss visibly recorded on the calendar (title prefix
+ box-shadow ring) rather than the slot silently freeing.

Output: A new `CampaignRunDecisionView._set_run_status()` branch with a server-side business-logic
guard and a staleness-safe conditional update; a guarded in-place `CalendarEvent` update that never
fabricates an event for a run that never had one; a new Decided-table action gated by an
independent `status_actions` flag; a one-line `_TERMINAL_PREFIXES` addition; and tests. No new model
field, no migration, no change to `_project_calendar_event()`.
</objective>
