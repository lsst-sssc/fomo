<objective>
Build the `sync_lco_observation_calendar` Django management command that, given
`--proposal <code>`, queries `ObservationRecord(facility='LCO',
parameters__proposal=<code>)` and upserts one `CalendarEvent` per matching record
— keyed on the real LCO portal URL — transitioning each event from an
unscheduled `[QUEUED]` banner to a clean placed block as the LCO scheduler acts,
and prefixing the title on terminal failure states while retaining the event.

Purpose: Stage 3 of the telescope-runs-calendar feature (issue #37) — gives users
a single command that keeps the FOMO calendar in sync with the LCO queue.
Output: One new management command file and one new test file; no new models or
migrations (CalendarEvent already has the required fields).

## Locked Decisions (NON-NEGOTIABLE)

- **D-01:** Build `url` via `LCOFacility().get_observation_url(record.observation_id)`
  (returns `https://observe.lco.global/requests/<id>`, no trailing slash). NEVER
  hardcode `requestgroups/<id>/` — that 404s. `url` is the upsert key.
- **D-02:** Derive `telescope` from a module-level `SITE_TELESCOPE_MAP` keyed on
  `parameters['site']` (LCO 3-letter site code). `instrument` is
  `parameters['instrument_type']` verbatim.
- **D-03:** Title is `'[QUEUED] {telescope} {instrument}'` while `scheduled_start`
  is None; clean `'{telescope} {instrument}'` once placed.
- **D-04:** Terminal-failure prefixes take priority over `[QUEUED]` — a record in
  a failure state shows only the terminal prefix, never both.
- **D-05:** `description` contains, in order: proposal code, record `status`, and
  the active time window (parameters start/end while unscheduled, scheduled
  start/end once placed).
- **D-06 (research correction — RESOLVED HERE):** `get_terminal_observing_states()`
  returns **5** states including `COMPLETED`, which TERM-01's prefix table has no
  entry for. This plan LOCKS the decision: use
  `LCOFacility().get_failed_observing_states()` (the 4 failure states) as the
  prefix trigger; `COMPLETED` records receive a **clean, no-prefix title** (same
  as a normally-placed record). This corrects the implicit 4-state assumption in
  CONTEXT.md D-04 / TERM-01 with an explicit, stated decision and a test.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Operator shell → command | `--proposal <code>` is free-text CLI input used directly in an ORM filter |
| ObservationRecord.parameters (third-party-populated JSON) → CalendarEvent fields | `parameters` values (proposal/start/end/instrument_type/site) are populated by the TOM submission layer and flow into CalendarEvent title/description/times/url |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering (injection) | `--proposal` → `parameters__proposal=code` ORM filter | accept | Django ORM parameterizes JSONField key-value lookups (value is bound, not string-interpolated — confirmed in RESEARCH.md generated SQL). No hand-rolled escaping. ASVS V5 satisfied by framework. |
| T-04-02 | Tampering / Information Disclosure | `parameters['start']`/`['end']`/`['site']`/`['instrument_type']` flowing into CalendarEvent fields | mitigate | Malformed/missing keys raise KeyError/ValueError caught per-record and skipped (logged by observation_id) — a bad record cannot corrupt the run or other events; datetime parsing via stdlib `fromisoformat` rejects malformed time strings into the skip path. |
| T-04-03 | Tampering | `url` field used as upsert key, derived from `observation_id` | mitigate | `url` built via `LCOFacility().get_observation_url()` (library helper, pure string-build), not from raw user input — prevents a crafted parameters value from redirecting the upsert key. |
| T-04-04 | Repudiation | Silent no-op run (zero matches) appears successful | mitigate | Always emit stdout summary with explicit `created: 0` counts so an operator sees a no-op run (RESEARCH.md security note). |
| T-04-05 | Denial of Service (low) | Very large matching result set | accept | Single-operator CLI tool, local SQLite, bounded by one proposal's records; no untrusted remote trigger. Low value/low likelihood. |
| T-04-SC | Tampering | npm/pip/cargo installs | accept | No new packages installed this phase (RESEARCH.md Package Legitimacy Audit: N/A — all classes from already-installed `tomtoolkit`). No legitimacy checkpoint required. |

No high-severity threats — security_block_on=high is satisfied (no blocking threats).
