<objective>
Add a one-off Django management command `backfill_range_calendar_events` that retroactively projects
CalendarEvents for already-APPROVED, site-resolved range-window CampaignRuns that were approved under
the old zero-event behavior — the real GS-2026A-FT-115 row (CampaignRun pk=34) being the motivating
case — plus a test suite proving it (D-07/FIX-08).

Purpose: Projection only fires on the approve / resolve_site POST actions, never retroactively, so
without this command every run approved before this phase (including pk=34) would stay eventless
forever. A management command (not a data migration) is D-07's locked choice: it is re-runnable,
idempotent, and keeps side-effecting projection logic out of migrations.

Output: `solsys_code/management/commands/backfill_range_calendar_events.py` (with a `--dry-run` flag)
and `solsys_code/tests/test_backfill_range_calendar_events.py`.

Decision locked this plan (D-07 was Claude's Discretion on name/location/flags): command name
`backfill_range_calendar_events`; `--dry-run` flag INCLUDED. Rationale: it writes real production
CalendarEvent rows against the live dev DB for the real pk=34 row, so a preview mode is standard
safety for a one-off data-writing command. No `--dry-run` precedent exists elsewhere in this
codebase (RESEARCH confirmed via grep); this is a deliberate, stated addition, not an accident.
</objective>

    - `dry_run = options['dry_run']`.
    - Query candidates: `CampaignRun.objects.filter(approval_status=CampaignRun.ApprovalStatus.APPROVED,
      site__isnull=False, window_start__isnull=False).exclude(window_start=F('window_end'))`. This
      selects APPROVED, site-resolved, range-window runs (window_start and window_end both non-null and
      unequal). NOTE: the query is deliberately not filtered by site observations_type — D-07's
      "ground-based site" describes the real motivating data (pk=34), and _project_calendar_event()
      already routes ground vs satellite correctly, so a hypothetical satellite range candidate is
      handled safely too. State this in a code comment.
    - Initialize counters: `backfilled_count = 0`, `skipped_count = 0`, `failed_count = 0`,
      `would_backfill_count = 0`.
    - For each `run` in candidates:
      * Existence pre-check: `already = CalendarEvent.objects.filter(Q(url=f'CAMPAIGN:{run.pk}') |
        Q(url__startswith=f'CAMPAIGN:{run.pk}:')).exists()` (trailing colon — Pitfall 2). If `already`:
        `skipped_count += 1`; `continue`.
      * If `dry_run`: write a line to self.stdout naming the run pk, campaign, telescope_instrument, and
        window (window_start..window_end) that WOULD be backfilled; `would_backfill_count += 1`;
        `continue` (write nothing).

        `ValueError as exc` (e.g. a resolved site with a blank timezone — the CR-01 re-raise from the
        ground branch): `logger.debug(...)`, write the failure to self.stderr naming the run pk and exc,
        `failed_count += 1`, `continue` — do NOT let one bad candidate abort the whole run.
    - Final summary via self.stdout.write: report candidate total and the counters. For a dry run,
      report `would_backfill` and `skipped`; for a real run, report `backfilled`, `skipped`, `failed`.
      Mirror load_telescope_runs.py's summary style.
