### Proposal Palette (DISPLAY-04)

**8-color curated palette, colorblind-vetted, white-text contrast ≥ 4.5:1 WCAG AA.**

Palette is project-local (not imported from `tom_calendar.utils.BOOTSTRAP_COLORS`) to survive the
pending `tomtoolkit` Bootstrap 5 rename (`var(--red)` → `var(--bs-red)`).

| Index | Hex | Hue family | Contrast with white | Notes |
|-------|-----|-----------|---------------------|-------|
| 0 | `#005f9e` | Blue (~215°) | ≥ 6.2:1 | Deuteranopia-safe: appears blue |
| 1 | `#a34000` | Burnt orange (~25°) | ≥ 5.5:1 | Deuteranopia-safe: appears brownish-orange |
| 2 | `#5b2080` | Deep violet (~270°) | ≥ 8.0:1 | Safe across all CVD types |
| 3 | `#006b4e` | Forest teal-green (~155°) | ≥ 5.4:1 | Different luminance from crimson in deuteranopia |
| 4 | `#9e1c1c` | Dark crimson (~0°) | ≥ 8.0:1 | Safe across all CVD types |
| 5 | `#006b6b` | Dark teal/cyan (~180°) | ≥ 5.3:1 | Grayish-blue in deuteranopia; distinct from blue |
| 6 | `#6b2060` | Dark magenta (~310°) | ≥ 7.5:1 | Safe across all CVD types |
| 7 | `#7a4500` | Dark sienna-brown (~35°) | ≥ 6.5:1 | Distinct luminance from other warm hues |

**Implementation mandate**: Normalize proposal string (`.strip().upper()`) before hashing. Hash via
`hashlib.sha256(normalized.encode()).hexdigest()`. Index into palette via
`int(digest, 16) % len(PALETTE)`. Empty string after normalization routes to the Neutral Slot (below),
not into the palette hash.

**CVD verification gate**: Before finalizing these 8 hex values in code, run them through a
deuteranopia/protanopia simulator (e.g. Coblis or Chromatic Vision Simulator) to confirm no two palette
entries are visually identical under the two most common deficiency types. If a conflict is found,
substitute the lighter-hued of the two conflicting entries with an adjacent hue from the same family
(e.g. shift from `#006b4e` toward `#00587a`). This is an implementation-time check, not pre-decided here.
The 8 entries above are the recommended starting point.

### Status Treatment (DISPLAY-05/06)

**Mechanism deferred to `/gsd:sketch` session per D-08.** The table below is the option set
narrowed by D-09 (dashed border excluded), with a clear recommendation. The sketch session picks one
and that choice becomes the canonical value in this section.

| Option | CSS property | Queued | Placed | Terminal-failure | Constraint check |
|--------|-------------|--------|--------|-----------------|-----------------|
| **box-shadow ring (RECOMMENDED)** | `box-shadow` | `0 0 0 2px rgba(0,0,0,0.45)` | none | `0 0 0 3px rgba(160,0,0,0.55)` | Different property from Phase 8's `border: 2px dashed` — composes cleanly, no collision |
| Solid border thickness | `border` shorthand | `border: 2px solid rgba(0,0,0,0.4)` | `border: 1px solid transparent` | `border: 3px solid rgba(160,0,0,0.5)` | RISK: `border` shorthand on same element as Phase 8's `border: 2px dashed`; whichever is last in the style string wins — fragile composition |
| Opacity reduction | `opacity` | `opacity: 0.65` | `opacity: 1` | `opacity: 0.45` | RISK: reduces both color AND text readability simultaneously; loses the proposal-color identity signal for terminal events |
| Stripe/hatching | `background-image` | `repeating-linear-gradient(...)` | none | different gradient | RISK: competes with `background-color` fill; visually noisy at the ~18-char event-block width per REQUIREMENTS.md Out of Scope |

**Pre-sketch recommendation (box-shadow):**

Because `box-shadow` is an independent CSS property from `border`, it composes with Phase 8's
`border: 2px dashed rgba(0,0,0,0.65)` without conflict. Both can be present simultaneously on the
same element, which is the correct behavior when an event is both fallback-labeled (Phase 8 cue) and
queued or terminal-failure (Phase 9 cue).

Status buckets and their CSS (pending sketch confirmation):

| Status bucket | Title prefix(es) | Recommended CSS |
|--------------|-----------------|-----------------|
| Queued | `[QUEUED] ` | `box-shadow: 0 0 0 2px rgba(0, 0, 0, 0.45);` |
| Placed — clean | no prefix | *(no status modifier)* |
| Placed — unverified | `[UNVERIFIED] ` | *(no status modifier — Phase 8's dashed border already covers this)* |
| Terminal failure | `[EXPIRED] ` / `[CANCELLED] ` / `[FAILED] ` | `box-shadow: 0 0 0 3px rgba(160, 0, 0, 0.55);` |

Note: "Placed" maps to two sub-states (`[UNVERIFIED]` and clean). DISPLAY-06's border-treatment treats
them identically (both = "placed") — Phase 8's separate dashed-border cue handles the
verified/unverified sub-distinction. A single status tag must not re-encode that distinction or the two
phases' signals overlap on the same visual fact.

## Interaction States

### Event Color (DISPLAY-04)

| State | `background-color` | Notes |
|-------|--------------------|-------|
| Default (all-day event) | `{% proposal_color event.proposal %}` output | Replaces `{{ event.color }}` at every existing call site |
| Default (timed event bullet) | `{% proposal_color event.proposal %}` output | Applied to `▌` bullet's `color:` inline style; timed event div itself has no background fill |
| Off-month day cells | `var(--light)` | Unchanged — not event elements |

### Status Modifier (DISPLAY-05/06)

| State | Additional CSS | Applied via |
|-------|---------------|------------|
| Queued | `box-shadow: 0 0 0 2px rgba(0,0,0,0.45)` | `status_border_css` template tag output |
| Placed | *(nothing)* | `status_border_css` returns empty string |
| Terminal-failure | `box-shadow: 0 0 0 3px rgba(160,0,0,0.55)` | `status_border_css` template tag output |
| Phase 8 fallback cue (unchanged) | `border: 2px dashed rgba(0,0,0,0.65)` | Existing `is_verified == False` check — composed alongside, not replaced |

**Composition rule**: The `style` attribute on each event div concatenates `background-color`,
`status_border_css` output, and (if `is_verified == False`) the dashed-border literal. All three are
independent CSS properties. No `{% elif %}` chain is used that would make them mutually exclusive.

### Legend / Click-to-Filter (DISPLAY-07)

| Interaction | DOM change | CSS result |
|-------------|-----------|-----------|
| No filter active | *(none)* | All `.cal-event` elements at default opacity |
| Click legend swatch (proposal X) | Add `.cal-filtering` on `#calendar-partial`; add `.cal-filter-match` on events where `data-proposal == X`; add `.is-active` on clicked swatch | Non-matching events: `opacity: 0.18; filter: grayscale(50%)`. Matching events: full opacity. Active swatch: `font-weight: 700; text-decoration: underline`. |
| Click same legend swatch again (toggle off) | Remove `.cal-filtering`; remove `.cal-filter-match` from all events; remove `.is-active` from all swatches | All events restored to default opacity |
| Click a different legend swatch (switch) | Move `.cal-filtering`, update `.cal-filter-match` set, move `.is-active` | Previous proposal dims, new proposal highlights |

**CSS rules to add** (inside the existing `<style>` block in `calendar.html`):

```css
.cal-legend-swatch {
  cursor: pointer;
  user-select: none;
  font-size: 0.7rem;      /* matches .cal-event-timed .cal-event-bullet */
  line-height: 1;
  transition: opacity 0.15s;
}
.cal-legend-swatch.is-active {
  font-weight: 700;
  text-decoration: underline;
}
.cal-event {
  transition: opacity 0.2s, filter 0.2s;
}
#calendar-partial.cal-filtering .cal-event {
  opacity: 0.18;
  filter: grayscale(50%);
}
#calendar-partial.cal-filtering .cal-event.cal-filter-match {
  opacity: 1;
  filter: none;
}
```

**JS mechanism** (small inline IIFE, `<script>` block at the end of `calendar.html` inside
`#calendar-partial` — re-executes on each htmx month swap, which resets filter state correctly):

```javascript
(function () {
  var container = document.getElementById('calendar-partial');
  if (!container) return;
  var activeProposal = null;
  container.addEventListener('click', function (e) {
    var swatch = e.target.closest('.cal-legend-swatch');
    if (!swatch) return;
    var proposal = swatch.dataset.proposal;
    if (activeProposal === proposal) {
      activeProposal = null;
      container.classList.remove('cal-filtering');
      container.querySelectorAll('.cal-event').forEach(function (el) { el.classList.remove('cal-filter-match'); });
      container.querySelectorAll('.cal-legend-swatch').forEach(function (el) { el.classList.remove('is-active'); });
    } else {
      activeProposal = proposal;
      container.classList.add('cal-filtering');
      container.querySelectorAll('.cal-event').forEach(function (el) {
        el.classList.toggle('cal-filter-match', (el.dataset.proposal || '') === proposal);
      });
      container.querySelectorAll('.cal-legend-swatch').forEach(function (el) {
        el.classList.toggle('is-active', (el.dataset.proposal || '') === proposal);
      });
    }
  });
}());
```

**htmx survival**: The IIFE is inside the `#calendar-partial` fragment. htmx `hx-swap="outerHTML"`
replaces this fragment on Prev/Next/Today clicks, which causes the new `<script>` to re-execute
(htmx re-evaluates scripts in swapped content by default). `activeProposal` is reset to `null`
on each page load/swap, which is correct — the filter does not survive month navigation, matching user
expectation (new month, new set of proposals).

**`data-proposal` attribute contract**: Each `.cal-event` div gets
`data-proposal="{{ event.proposal|upper|strip }}"` so the JS comparison matches the normalized key
used by `visible_proposals`. Each `.cal-legend-swatch` gets `data-proposal="{{ entry.key }}"` where
`entry.key` is the normalized proposal string (or the empty string for the Classical schedule entry).

## Legend Layout

The legend lives in the existing footer row (`d-flex justify-content-between align-items-center mt-1`),
in the left `<div>` alongside the target-list swatches. It follows the exact swatch markup pattern
confirmed live from `target_list_block.html` (D-01):

```django
{# Source: D-01 -- mirrors target_list_block.html swatch pattern exactly #}
{% for entry in proposal_legend %}
  <span class="cal-legend-swatch mr-2"
        role="button"
        aria-pressed="false"
        data-proposal="{{ entry.key }}"
        style="color: {{ entry.color }};">&#x258C;</span>
  <small class="text-secondary mr-2">{{ entry.label }}</small>
{% endfor %}
```

- `entry.key` — normalized proposal string (`.strip().upper()`) or `""` for Classical schedule
- `entry.label` — comma-separated sorted proposal codes, or `"Classical schedule"` for the neutral entry
- `entry.color` — hex from `proposal_color()` output, or `#5a6268` for Classical schedule

Collision grouping: if two proposals hash to the same color, `entry.label` is `"ProposalA, ProposalB"`
(all colliding codes under one swatch). The `data-proposal` attribute uses the first (alphabetically
sorted) code as the filter key — events for both proposals share this key in their `data-proposal`
attribute, so clicking the combined swatch highlights both.

Ordering: legend entries sorted by color hex string (stable, deterministic across re-renders).
Classical schedule entry (neutral slot) always appears last.
