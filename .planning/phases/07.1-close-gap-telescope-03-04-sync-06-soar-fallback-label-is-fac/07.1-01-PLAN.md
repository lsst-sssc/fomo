<objective>
Close the v1.3 milestone-audit gap in TELESCOPE-03/TELESCOPE-04/SYNC-06: the coarse-label
fallback path in `sync_lco_observation_calendar.py` is facility-unaware. `_coarse_telescope_label`
assumes every `instrument_type` is LCO-style aperture-class-prefixed (`'1M0-SCICAM-SINISTRO'`),
so a SOAR instrument type (`'SOAR_GHTS_REDCAM'`) never matches and the function returns the raw
instrument string. Combined with `_title_for`'s `f'[UNVERIFIED] {telescope} {instrument}'` template,
a placed SOAR record whose live API call fails produces the doubled, non-coarse title
`'[UNVERIFIED] SOAR_GHTS_REDCAM SOAR_GHTS_REDCAM'` instead of a clean coarse `4m0` label.

Purpose: Give a FOMO operator a correct, usable degraded label for a SOAR record that hit the
fallback path, matching what TELESCOPE-03/04/SYNC-06 specify. SOAR has exactly one site and one
aperture class, so the correct fallback is unconditionally `4m0` (per the existing single
`('sor','4m0'): 'SOR-4m0'` entry in `SITE_TELESCOPE_MAP`).

Output: A facility-aware `_coarse_telescope_label`, a new test driving a placed `facility='SOAR'`
record through the API-failure fallback (closing the audit's zero-coverage gap for
SOAR+placed+API-failure), and an updated demo notebook cell showing the corrected label.
</objective>

no new user-input parsing. It consumes data already in memory (`record.facility`, `instrument_type`)
and emits a short, fixed-vocabulary token (`'4m0'`) or the unchanged LCO logic. ASVS L1 surface for
this change is therefore minimal; threats are enumerated and dispositioned explicitly rather than
omitted.
