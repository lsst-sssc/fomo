# Phase 8 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope note:** This phase is a server-rendered Django/Bootstrap 4 template change (one conditional
branch added twice to an existing full-copy override of `tom_calendar`'s `calendar.html`), not a
React/SPA build. There is no `package.json`, `components.json`, or Tailwind config in this repo — the
shadcn initialization gate from the standard UI-researcher flow does not apply. This contract adapts the
same rigor (spacing, color, typography, copywriting, accessibility) to the actual stack: Bootstrap 4
utility classes + inline `style=` attributes + Django template conditionals, matching the existing
`[QUEUED]` override precedent at `calendar.html:158-162`.

## Spacing Scale

This phase adds zero new spacing/layout — the dashed-border treatment reuses the exact box model of the
existing `.cal-event-all-day` / `.cal-event-timed` elements (no new padding, margin, or gap values).

| Token | Value | Usage in this phase |
|-------|-------|----------------------|
| border-width (verified) | 1px | Existing default border (no border declared in the `{% else %}`/unstyled branch today — solid/default look is unchanged) |
| border-width (fallback) | 2px | New dashed border — matches the existing `[QUEUED]` override's `1px solid` weight class, bumped to 2px so the dash pattern is visually legible at the ~16-18 char event-block width |

Exceptions: none — no 4px-multiple spacing token is introduced; this phase is a border-style/attribute
change only, not a layout change.

## Color

This phase does not touch `CalendarEvent.color` (proposal-keyed palette is Phase 9's DISPLAY-04/05/06).
The only new visual signal is a border treatment layered on top of whatever `background-color` the event
already renders with.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | inherited Bootstrap 4 page background (`#fff`/`var(--light): #f8f9fa`) | Calendar grid background, day cells — unchanged by this phase |
| Secondary (30%) | `var(--secondary): #6c757d` | Existing day-header/secondary text, `[QUEUED]` grey override (`rgba(0,0,0,0.45)` background / `rgba(0,0,0,0.55)` border) — unchanged by this phase |
| Accent (10%) | `var(--primary): #007bff` | Reserved for: "Today" date-circle highlight (`.today .day-num`) only — this phase introduces no new accent-color usage |
| Verification cue (new, this phase) | `rgba(0, 0, 0, 0.65)` border color, 2px **dashed** style | Fallback-labeled events only (`event.telescope_label_meta.is_verified == False`), both `.cal-event-all-day` and `.cal-event-timed` branches. Verified events and events with no sidecar row get no border change (today's default, unstyled/inherited border). |
| Destructive | not applicable | No destructive action in this phase's scope (no delete/remove UI added) |

Accent reserved for: the existing "Today" date-circle background only (`.cal-day.is-current-month.today
.day-num`) — this phase adds no new accent-color element. The new dashed-border color
(`rgba(0,0,0,0.65)`) is a neutral grey, deliberately **not** drawn from the primary/accent slot, so it
does not visually compete with Phase 9's future proposal-color palette sitting in the same
`background-color` underneath it.

**Border-property reservation (D-03, locked):** dash-style is reserved exclusively for this phase's
verification signal. Phase 9's status visual treatment (queued/placed/terminal-failure) MUST use a
different border property — color, thickness, or double-border — never dash-style. This is a hard
constraint carried into Phase 9's own UI-SPEC/CONTEXT, not optional.

## Interaction Contract

(Non-template fields the standard contract doesn't have a slot for, called out explicitly since this
phase is interaction-cue-only.)

| Behavior | Spec |
|----------|------|
| Visual cue trigger | `{% if event.telescope_label_meta.is_verified == False %}` — explicit `False` comparison, never bare truthiness (per RESEARCH.md Pattern 2, deliberately makes "no sidecar row" and "explicit `True`" both fall through to the unstyled/verified branch) |
| Visual cue rendering | 2px dashed border (`rgba(0, 0, 0, 0.65)`), applied identically to `.cal-event-all-day` and `.cal-event-timed` wrapping `<div>` — same element that already carries the `[QUEUED]` override and the title text, per the existing precedent |
| Tooltip trigger | Native HTML `title="..."` attribute, placed on the same wrapping `<div>` as the dashed-border style (not a child element) — must be on the hoverable element, not buried inside a flex child that doesn't receive `:hover` (Pitfall 4) |
| Tooltip mechanism | Native browser tooltip only — no JS tooltip library, no Bootstrap `data-toggle="tooltip"` (would require JS init this phase doesn't add) |
| Discoverability requirement | Both the dashed border AND the tooltip must ship together in the same task, on the same element — DISPLAY-02 (visual cue) and DISPLAY-03 (tooltip) are separate success criteria but must not ship independently (Pitfall 4) |
| `[QUEUED]` precedence | The existing `{% if event.title|slice:":9" == "[QUEUED] " %}` branch takes precedence (checked first, per the existing `{% if %}/{% elif %}/{% else %}` chain) — a queued + fallback-labeled event renders with the `[QUEUED]` grey treatment in this phase; reconciling the two signals visually is explicitly Phase 9's concern (DISPLAY-05/06), not this phase's |
| Accessibility fallback | The existing `[UNVERIFIED]` title-text prefix (shipped Phase 07/07.1) remains unchanged and is the WCAG 1.4.1-satisfying non-color/shape-only channel — this phase's border + tooltip are additive, not a replacement |
