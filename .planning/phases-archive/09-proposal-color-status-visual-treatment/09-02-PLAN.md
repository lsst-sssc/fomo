<resolved_decisions>
Two contradictions exist between source artifacts. This plan RESOLVES them — implement as stated
here, not as the literal snippets in 09-PATTERNS.md / 09-UI-SPEC.md:

1. **`data-proposal` is keyed on the resolved COLOR hex, not the raw proposal string.**
   09-UI-SPEC.md's legend snippet uses `data-proposal="{{ entry.key }}"` and an event snippet
   `data-proposal="{{ event.proposal|upper|strip }}"`. Two problems: `strip` is NOT a Django filter
   (template parse error), and the locked click-to-filter JS uses strict `===` equality, which
   cannot group two hash-colliding proposals under one swatch (D-04) if keyed on the raw code.
   Resolution: set `data-proposal="{% proposal_color event.proposal %}"` on each `.cal-event`
   element and `data-proposal="{{ entry.color }}"` on each legend swatch. Colliding proposals share
   a color → share a key → highlight together. The locked JS is used UNCHANGED.

2. **`data-proposal` lives on the `.cal-event` element the JS targets.** The filter JS selects
   `.cal-event` descendants. For all-day events that is the OUTER `div.cal-event.cal-event-all-day-row`
   (NOT the inner `.cal-event-all-day` styled pill). For timed events it is the
   `div.cal-event.cal-event-timed`. Put `data-proposal` there.

3. **Status mechanism = box-shadow (D-08 resolved in 09-UI-SPEC.md).** Ignore 09-PATTERNS.md's
   `border: 3px solid …` placeholders. The exact strings come from Plan 01's `status_border_css`.
</resolved_decisions>

| T-09-01 | Tampering | `style="background-color: {{ bg_color }}; {{ status_border }}"` interpolation | high | mitigate | `bg_color` and `status_border` are NEVER the raw `proposal`/`title` string — they are the fixed-palette/box-shadow-literal outputs of Plan 01's tags. The raw field is used only as hash input / startswith test (T-09-01 control lives in Plan 01). This template interpolates only those computed values into `style`; it never puts `{{ event.proposal }}` or `{{ event.title }}` raw into a style attribute. |
| T-09-02 | Tampering | legend `{{ entry.label }}` and `data-proposal="{{ entry.color }}"` | medium | mitigate | `entry.color` is a fixed-palette hex; `entry.label` is rendered through Django's default autoescaping (no `|safe`, no `mark_safe`). `data-proposal` values are computed colors, not echoed free text. |
