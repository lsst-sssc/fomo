<objective>
Add a pure-Python classical-schedule run-line parser to the existing
`solsys_code/telescope_runs.py`: a `ParsedRun` dataclass and a
`parse_run_line(line: str) -> ParsedRun` function that turns a free-text run
line into structured (telescope, instrument, status, year, month, day1, day2)
fields. This is the input contract Phase 3's `load_telescope_runs` command
consumes.

Purpose: Phase 3 cannot expand runs into CalendarEvents until run lines are
parsed into structured fields. `ParsedRun.telescope` (a resolved SITES key)
feeds directly into Phase 3's `get_site(parsed.telescope)`.

Output: `ParsedRun`, `parse_run_line()`, and a `KNOWN_STATUSES` constant in
`solsys_code/telescope_runs.py`, plus parser tests in
`solsys_code/tests/test_telescope_runs.py`.
</objective>

       - Telescope resolution (D-01): take the first whitespace-delimited
         token as the telescope candidate. Resolve by PREFIX match against
         SITES.keys() (the four keys: 'Magellan-Clay', 'Magellan-Baade',

         them (per CONTEXT.md <specifics> for the Proto-Lightspeed sample) —

         bare word and inside parentheses (e.g. '(proposed)', '(not
         confirmed)'). Match case-insensitively against KNOWN_STATUSES,
         supporting multi-word phrases. If a status-shaped token/parenthesized
         phrase is present but NOT in KNOWN_STATUSES, raise ValueError (D-06).
         If no status is present, default status='allocation' (D-05). Store
         the matched status in its KNOWN_STATUSES (lowercase) form.

       - Instrument (D-03/PARSE-02): the token(s) between the telescope token
         and the status/date portion. A hyphenated instrument such as
         'Proto-Lightspeed' MUST parse as a single token — do not split on the
         hyphen. (The date-range hyphen in '9-13' is digits-hyphen-digits;
         keep the instrument-vs-date distinction unambiguous in the regex.)

       - Date range (PARSE-01): support BOTH orderings via two regex patterns

         views.py:_translate_constraints precedent):

             '9-13 July' -> day1=9, day2=13, month=July
           * month-after-range: 'Jul 8-12' -> month=July, day1=8, day2=12
         Accept both full month names and 3-letter abbreviations,
         case-insensitively; convert to month number 1-12. Also support a
         cross-month/cross-year range where the start and end are in different
         months (e.g. '28 December-2 January'): month is the month of day1.

       - Year (PARSE-03): no year token is present in the sample lines, so
