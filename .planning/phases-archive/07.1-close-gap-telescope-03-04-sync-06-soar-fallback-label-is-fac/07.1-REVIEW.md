# Phase 07.1: Code Review Report

**Reviewed:** 2026-06-24T15:06:20Z
**Depth:** standard
**Files Reviewed:** 3
**Status:** issues_found

## Summary

This phase makes `_coarse_telescope_label` facility-aware so SOAR records get an
unconditional `'4m0'` fallback label instead of falling through to the raw,
unmatched `instrument_type` string (which previously produced a doubled
`'[UNVERIFIED] SOAR_GHTS_REDCAM SOAR_GHTS_REDCAM'` title for a placed SOAR
record whose live API resolution failed).

I traced the three risk areas called out in the prompt:

- **SOAR-detection predicate**: `facility_name.upper() == 'SOAR'` is an exact
  equality check (not substring/`in`), case-normalized via `.upper()`. No false
  positives possible for facility strings like `'SOARX'`. Verified that
  `record.facility` is only ever exactly `'LCO'` or `'SOAR'` at this call site,
  because the only path into `_build_event_fields` (and therefore into
  `_coarse_telescope_label`) is gated by `facility = facilities.get(record.facility)`
  returning non-`None`, which itself only succeeds for the dict keys `'LCO'`/`'SOAR'`,
  which in turn are populated only from `ObservationRecord.objects.filter(facility__in=['LCO', 'SOAR'])`
  (a case-sensitive Django `__in` filter). So `.upper()` here is defensive but
  never actually fires on anything but already-uppercase input — harmless, not a bug.
- **LCO branch left genuinely untouched**: confirmed via `git diff` against the
  pre-change commit — the `if len(instrument_type) >= 3: ... return candidate`
  block and the final `return instrument_type` are byte-for-byte identical to
  before; only a new `if facility_name.upper() == 'SOAR': return '4m0'` guard was
  prepended, plus signature/docstring updates.
- **Call-site parameter mix-up**: `_coarse_telescope_label(instrument, record.facility)`
  passes `record.facility` (the model's plain string field, `'LCO'`/`'SOAR'`),
  not the `facility` parameter in scope (an `LCOFacility`/`SOARFacility`
  *instance*). This is correct — confirmed no mix-up. Calling `.upper()` on an
  `LCOFacility` instance would have raised `AttributeError`, and no test masks
  this because the new regression test goes through `call_command` end-to-end
  (a mix-up would have crashed the whole test, not just produced a wrong label).

I also reran the full Django test suite for this file (36/36 pass) and `ruff
check`/`ruff format --check` (clean) to confirm no regressions.

The regression test and demo notebook addition are well-targeted at the
highest-risk path (placed SOAR record + API failure → fallback), which is
exactly the v1.3 milestone-audit zero-coverage gap. One coverage gap remains
(see WR-01): the *banner-stage* SOAR call site of `_coarse_telescope_label`
(scheduled_start is None) is exercised incidentally by several pre-existing
tests but never explicitly asserted to produce `'4m0'`.

## Warnings

### WR-01: Banner-stage SOAR fallback label is never explicitly asserted

**File:** `solsys_code/tests/test_sync_lco_observation_calendar.py`
**Issue:** `_coarse_telescope_label(instrument, record.facility)` is called
unconditionally in `_build_event_fields` before the `scheduled_start is None`
branch (sync_lco_observation_calendar.py:420), so it also resolves the
fallback label for banner-stage (unscheduled) SOAR records, not just placed
ones. Several existing tests create banner-stage SOAR records (e.g.
`test_select_03_all_token_case_insensitive_syncs_everything` at line 469,
`test_select_04_single_run_covers_both_facilities` at line 488,
`test_select_05_soar_record_uses_soar_facility_instance` at line 519,
`test_extract_02_soar_multi_config_picks_spectrum_not_calibration` at line
554) but none of them assert `event.telescope == '4m0'` or check the title for
the doubled-instrument defect on that path — they only assert that an event
was created or that `event.instrument` is correct. The only test that
explicitly asserts the `'4m0'` coarse label is
`test_telescope_03_soar_api_failure_fallback_returns_4m0_label`, which is
*placed*-only (`scheduled_start` populated, API timeout). If a future change
accidentally reintroduced facility-unaware behavior on the banner-stage branch
only (e.g. by special-casing on `record.scheduled_start is None` instead of
calling the same helper), no existing test would catch it.
**Fix:** Add a banner-stage assertion, e.g. extend
`test_select_04_single_run_covers_both_facilities` (or add a small dedicated
test) to assert `CalendarEvent.objects.get(url=soar_url).telescope == '4m0'`
and that the title doesn't contain `'SOAR_GHTS_REDCAM SOAR_GHTS_REDCAM'`,
mirroring the assertions already made for the placed path.

## Info

### IN-01: `_coarse_telescope_label` has no direct unit test

**File:** `solsys_code/tests/test_sync_lco_observation_calendar.py`
**Issue:** Unlike `_aperture_class_from_telescope_code`, `_derive_telescope`,
and `_resolve_placement_block` — which are imported directly and unit-tested
in isolation (lines 21-24, 673-711) — `_coarse_telescope_label` is never
imported into the test module and is only exercised indirectly through
`call_command`. This makes the SOAR/LCO branch logic harder to test
exhaustively (e.g. boundary cases like a 2-character `instrument_type`, an
empty string, or a facility value with mixed case such as `'Soar'`) without
spinning up a full command run with DB fixtures.
**Fix:** Import `_coarse_telescope_label` and add a small direct unit test,
e.g.:
```python
from solsys_code.management.commands.sync_lco_observation_calendar import (
    _coarse_telescope_label,
)

def test_coarse_telescope_label_soar_is_unconditional(self):
    self.assertEqual(_coarse_telescope_label('SOAR_GHTS_REDCAM', 'SOAR'), '4m0')
    self.assertEqual(_coarse_telescope_label('1M0-SCICAM-SINISTRO', 'SOAR'), '4m0')
    self.assertEqual(_coarse_telescope_label('soar-anything', 'soar'), '4m0')
    self.assertEqual(_coarse_telescope_label('1M0-SCICAM-SINISTRO', 'LCO'), '1m0')
    self.assertEqual(_coarse_telescope_label('UNKNOWN-XYZ', 'LCO'), 'UNKNOWN-XYZ')
```
