# Phase 15 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope:** VIEW-01..04 — a per-campaign `CampaignRun` table (sortable/paginated/filterable), a
campaigns list page, a target-detail integration point, and a navbar entry. Server-rendered
Django templates (TOM Toolkit / Bootstrap 4), **not** a JS/SPA component framework. The shadcn
gate does not apply — see Design System row below.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none — this is a server-rendered Django/Bootstrap 4 app (TOM Toolkit), not React/Next.js/Vite. The shadcn initialization gate is not applicable and was skipped. |
| Preset | not applicable |
| Component library | Bootstrap 4 (`bootstrap4` Django app, already project-wide) + `django-tables2` 3.0.0 with its bundled `django_tables2/bootstrap4-responsive.html` template + `django-filter` 24.3 form rendering. Both libraries are already installed/`INSTALLED_APPS`-registered per 15-RESEARCH.md; this phase is their first real consumer. |
| Icon library | Font Awesome 4.7.0 (`font-awesome-4.7.0/css/font-awesome.min.css`, already loaded globally in `tom_common/base.html`) |
| Font | System default via Bootstrap 4's reset — `src/static/tom_common/css/custom.css` sets `body { font-size: 1rem; font-weight: 400; line-height: 1.5; color: #212529; }`. No custom `@font-face`/web-font import exists anywhere in the project. This phase introduces no new font. |

## Spacing Scale

Bootstrap 4's `$spacer: 1rem` (16px) utility scale maps directly onto the 8-point scale. **Use
Bootstrap 4 spacing utility classes** (`p-*`, `m-*`, `mb-*`) — do not add new custom CSS for
layout spacing.

| Token | Value | Bootstrap 4 utility | Usage |
|-------|-------|----------------------|-------|
| xs | 4px | `.p-1` / `.m-1` (`.25rem`) | Badge internal padding, icon-to-text gaps |
| sm | 8px | `.p-2` / `.m-2` (`.5rem`) | Filter-field gaps, `table-sm` cell padding |
| md | 16px | `.p-3` / `.m-3` (`1rem`) | Default card/panel padding; gap between filter panel and table |
| lg | 24px | `.p-4` / `.mb-4` (`1.5rem`) | Page heading → filter panel; filter panel → table |
| xl | 32px | not used this phase | No multi-section landing layout needed at this scale |
| 2xl | 48px | `.p-5` (`3rem`) | Top-of-page margin, campaigns list page only |
| 3xl | 64px | not used this phase | No marketing/hero layout in this phase |

**Exceptions:** `CampaignRunTable.Meta.attrs = {'class': 'table table-bordered table-sm'}` uses
Bootstrap 4's `table-sm` modifier, which reduces cell padding to ~0.3rem (~5px) vertical — a
deliberate exception to the 8px-multiple rule. This is **required**, not stylistic drift: D-09's
spreadsheet-parity column set (13 columns for staff, 11 for anonymous — see Layout Contract
below) needs the density to stay usable inside `.table-responsive` without excessive horizontal
scroll.

## Typography

This phase introduces no new typography system — it reuses Bootstrap 4's existing defaults
project-wide. Only 3 sizes / 2 weights are needed for these new pages/partials:

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body / table cell text / free-text columns | 16px (`1rem`, Bootstrap 4 default) | 400 (regular) | 1.5 |
| Small / metadata (pagination "Page X of Y", filter helper text, run count) | 14px | 400 (regular) | 1.5 |
| Label / table header (`<thead><th>`) / badge text | 14px | 700 (bold — apply `.font-weight-bold` explicitly; do not rely on the browser UA-stylesheet default for `<th>` alone, for consistency across the badge/label family) | 1.2 |
| Page heading (e.g. "3I/ATLAS — Observing Runs") | 24px (`<h4>` element, styled with `.font-weight-bold`) | 700 | 1.2 |

**Note:** Bootstrap 4's default heading weight is 500 (`$headings-font-weight`), which is not one
of this contract's 2 declared weights. Add `.font-weight-bold` explicitly to the `<h4>` page
heading and to badge/label text rather than relying on bare heading-tag or `<th>` defaults, so
every bold element on this page is genuinely weight 700.

## Color

Reuses the palette already defined in `src/static/tom_common/css/custom.css` `:root` — **no new
hex values are introduced by this phase.**

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#ffffff` (`body { background-color: #fff; }`, already project-wide) | Page background, default table row background |
| Secondary (30%) | `#f8f9fa` (`--light`) | `thead-light` table header row, `table-striped` alternate rows, filter-panel card background (`bg-light`) |
| Accent (10%) | `#007bff` (`--primary`, matches existing `a { color: #007bff; }`) | Reserved for: campaign-name hyperlinks (campaigns list page), sortable-column header links (django-tables2 default sort-link styling), pagination active-page indicator, the filter panel's "Filter" submit button (`btn-primary`) |
| Destructive/Alert | `#dc3545` (`--danger`) | Reserved for: `approval_status = rejected` badge only (see Approval-Status Badge Contract below). **Not** used for any `run_status` value — see Run-Status Badge Contract for why. |

**Deliberate exception — target-detail button color:** The new campaign link button(s) (D-01/D-02)
reuse the existing `btn btn-info` (`#17a2b8`) class already used by the "Make Ephemeris" button
(`src/templates/solsys_code/partials/ephem_button.html`), per the explicit instruction to match
that established button-row precedent rather than introduce a new accent color into the same
button group. This is intentional consistency with pre-existing FOMO UI, not part of this phase's
fresh 60/30/10 accent allocation.

### Approval-Status Badge Contract (D-08 — mandatory)

`approval_status` gets a visually distinct, semantically-colored badge on every row (not a plain
column) so non-approved rows are spottable at a glance among approved ones (D-05/D-06: all rows,
regardless of status, are shown to everyone). Follow `calendar_display_extras.py`'s pattern
*shape* — fixed internal constant, badge class never derived from the raw DB string — but as a
plain 3-entry dict, not a hash-based palette (there are only 3 fixed values, not an unbounded set):

| `approval_status` value | Badge class | Badge color | Text color |
|--------------------------|-------------|-------------|------------|
| `pending_review` | `badge-warning` | `#ffc107` | `#212529` (Bootstrap 4 default dark-on-warning; meets WCAG AA) |
| `approved` | `badge-success` | `#28a745` | `#ffffff` |
| `rejected` | `badge-danger` | `#dc3545` | `#ffffff` |

Implement via a `render_approval_status` method on `CampaignRunTable` (per 15-RESEARCH.md Pitfall
2 — do not rely on automatic `get_FOO_display()`, since the non-staff `.values()` queryset yields
plain dict rows) returning `format_html('<span class="badge {}">{}</span>', badge_class, label)`.

### Run-Status Badge Contract (Claude's discretion — recommended for row-scanability)

`run_status` (8 values) also gets a badge for scan-ability, but deliberately **muted/neutral**
so it never visually competes with the mandatory approval_status badge in the same row:

| `run_status` value(s) | Badge class |
|------------------------|-------------|
| `requested`, `planned` | `badge-secondary` (`#6c757d`) |
| `observed`, `reduced` | `badge-info` (`#17a2b8`) |
| `published` | `badge-primary` (`#007bff`) — the run's natural positive end-state; the one deliberate use of accent blue as a badge outside the "reserved for" list above |
| `cancelled`, `not_awarded`, `weather_tech_failure` | `badge-light` with `border: 1px solid #6c757d` — **explicitly not `badge-danger`.** Using danger-red here would make an `approval_status=approved` + `run_status=cancelled` row look like the whole row was rejected, which is wrong: the approval was fine, only the real-world observation failed. Danger-red is reserved exclusively for `approval_status=rejected`. |

Implement via `render_run_status` on the same Table class, same reasoning as above (dict-row
compatibility).

## Layout Contract

### Page 1 — Campaigns list (D-03)

Plain `ListView` (no django-tables2 needed — no locked sort/filter/paginate requirement here per
15-RESEARCH.md Open Question 2), queryset `TargetList.objects.filter(campaign_runs__isnull=False)
.distinct().annotate(run_count=Count('campaign_runs'))`.

- Page heading: `<h4 class="font-weight-bold mb-4">Campaigns</h4>`
- Body: a simple Bootstrap 4 list (`<div class="list-group">` or a 2-column `<table class="table">`
  — planner's call, no locked decision requires table machinery), one row per campaign:
  - Campaign name as an accent-colored hyperlink to `campaigns:table` (its per-campaign table)
  - A muted `badge badge-secondary` showing `{{ run_count }} run{{ run_count|pluralize }}` next to
    the name (not a separate "View" button — see Copywriting Contract)
- Empty state (zero qualifying campaigns) — see Copywriting Contract

### Page 2 — Per-campaign table (VIEW-01, VIEW-04)

`CampaignRunTableView(SingleTableMixin, FilterView)`. Structure top to bottom:

1. **Page heading** (`<h4 class="font-weight-bold mb-4">`): `"{{ campaign.name }} — Observing
   Runs"` (e.g. "3I/ATLAS — Observing Runs")
2. **Filter panel** (`<div class="card bg-light p-3 mb-4">`), a single GET form containing:
   - `run_status`: rendered as `forms.CheckboxSelectMultiple` (NOT the django-filter default
     `SelectMultiple` — a native multi-select `<select multiple>` box is poor UX for 8 options).
     Lay out as inline Bootstrap 4 checkboxes (`form-check form-check-inline`) wrapping each of
     the 8 `RunStatus` labels.
   - `open_to_collaboration`: rendered as a 3-state `<select>` — "Any" / "Yes" / "No" (django-filter's
     default `BooleanFilter` widget produces Unknown/Yes/No; relabel "Unknown" → "Any" so the
     default D-07 unfiltered state reads naturally, not as an error state)
   - Submit button: `<button class="btn btn-primary">Filter</button>` (accent color, per Color
     contract)
   - Reset link: `<a class="btn btn-link">Clear filters</a>` linking to the bare table URL
     (no query string)
3. **Table** (`{% render_table table %}`), configured with:
   - `Table.Meta.template_name = 'django_tables2/bootstrap4-responsive.html'` — this wraps the
     table in `.table-container.table-responsive` automatically (confirmed present in the
     installed django-tables2 3.0.0 package), giving horizontal scroll for the wide column set
     without any custom CSS.
   - `Table.Meta.attrs = {'class': 'table table-bordered table-sm'}`
   - `Table.Meta.order_by = ('-obs_date',)` (D-10 default sort)
   - `table_pagination = {'per_page': 25}` (D-11) — the bundled Bootstrap 4 template renders
     Bootstrap 4's native pagination component automatically, no custom pagination markup needed.
   - Empty-table state (all rows filtered out) — see Copywriting Contract; django-tables2 renders
     this automatically via `Table.Meta.empty_text`, set to the Copywriting Contract's "No runs
     match these filters." body copy plus a "Clear filters" link.

### Column set (D-09 — spreadsheet-parity) and staff/anonymous visibility

| Column | Staff sees | Anonymous sees | Render notes |
|--------|:---:|:---:|--------------|
| `telescope_instrument` | Yes | Yes | plain text |
| `site` | Yes | Yes | `site.short_name` when `site` is set; when `site is None` and `site_needs_review=True`, render `site_raw` in italic/muted text (`text-muted`) with a small `fa-exclamation-triangle` icon and `title="Site could not be automatically resolved"` — mirrors the existing "flag, don't silently guess" pattern (`CalendarEventTelescopeLabel.is_verified`) already used elsewhere in FOMO |
| `obs_date` | Yes | Yes | default sort column (D-10), plain date |
| `ut_start` / `ut_end` | Yes | Yes | two columns, `HH:MM` UTC |
| `filters_bandpass` | Yes | Yes | plain text |
| `run_status` | Yes | Yes | badged, see Run-Status Badge Contract |
| `approval_status` | Yes | Yes | badged, see Approval-Status Badge Contract (D-08) — shown to everyone per D-05/D-06, not staff-only |
| `open_to_collaboration` | Yes | Yes | Yes/No text or `fa-check`/`fa-times` icon (planner's call; no locked decision requires an icon) |
| `contact_person` | Yes | **Omitted entirely** (D-13) | plain text, staff only |
| `contact_email` | Yes | **Omitted entirely** (D-13) | `mailto:` link, staff only |
| `observation_details`, `weather`, `observation_outcome`, `publication_plans`, `comments` | Yes | Yes | see Free-Text Column Truncation below |

**Column count:** 13 columns for staff, 11 for anonymous — this is a real structural difference in
the rendered `<table>` (fewer `<th>`/`<td>` per row), not a same-column-count masked/blanked
value, per D-13. Do not render an empty "Contact" header for anonymous users; the column must not
exist in the anonymous table's `Table.Meta.exclude` / restricted `.values()` field list at all.

**Free-text column truncation:** `observation_details`, `weather`, `observation_outcome`,
`publication_plans`, and `comments` can be long free-text. To avoid unusably wide/tall rows in a
13-column table, truncate each to 80 characters via `{{ value|truncatechars:80 }}` inside a `<td>`
styled with `white-space: nowrap; overflow: hidden; text-overflow: ellipsis; max-width: 200px;`
and set `title="{{ value }}"` on the `<td>` so the full text is available on hover. **Known
accessibility limitation, accepted for this phase:** `title` tooltips are not reliably reachable
by touch or keyboard-only navigation; a full "expand row" affordance is out of scope for VIEW-01's
locked requirement (spreadsheet-parity columns, not a detail-expansion pattern per D-09) and is
not required by any success criterion in this phase.

### Target-detail integration (D-01, D-02, VIEW-02)

Extend `SolsysCodeConfig.target_detail_buttons()` with a `campaign_links` entry rendered inside
the existing button row (`module_buttons.html`, unchanged — falls through to
`show_individual_app_partial`, confirmed in 15-RESEARCH.md Pattern 2).

- Zero matching campaigns: render nothing (empty partial output, no placeholder text) — matches
  the existing pattern where absent buttons simply don't appear.
- One matching campaign (today's case — 3I/ATLAS is single-target/single-campaign): a single
  `<a class="btn btn-info" href="...">View {{ campaign.name }} Runs</a>` — visually identical to
  the pre-existing "Make Ephemeris" button.
- Two or more matching campaigns (D-02, future-proofing): wrap in `<div class="d-flex
  flex-wrap">`, one `<a class="btn btn-info mr-2 mb-2">View {{ campaign.name }} Runs</a>` per
  campaign — `mr-2 mb-2` (8px gaps) ensures buttons wrap cleanly on narrow viewports without
  overlapping.

### Navbar integration (D-03, VIEW-02)

New `SolsysCodeConfig.nav_items()` entry, rendered via `navbar_app_addons` (empty-context call,
confirmed in 15-RESEARCH.md Pattern 3) — must match the existing navbar item markup exactly
(`tom_common/partials/navbar_app_addons.html` precedent), not introduce a new visual style:

```html
<li class="nav-item {% if request.resolver_match.namespace == 'campaigns' %}active{% endif %}">
    <a class="nav-link" href="{% url 'campaigns:list' %}">Campaigns</a>
</li>
```

Single word, sentence case, matching the existing "Targets" / "Data" / "Users" navbar entries — no
icon, no dropdown (this phase only needs one destination, the campaigns list page).

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA — target-detail integration (D-02, per matching campaign) | `"View {{ campaign.name }} Runs"` (e.g. "View 3I/ATLAS Runs") — not a generic "View Campaign," so 2+ buttons are distinguishable at a glance |
| Primary CTA — campaigns list → per-campaign table | The campaign name itself is the link (e.g. "3I/ATLAS"), with a `"{{ run_count }} run{{ run_count\|pluralize }}"` badge alongside — not a separate "View" button |
| Navbar entry (D-03) | `"Campaigns"` — matches the existing single-word navbar convention (Targets, Observations, Data, Users) |
| Filter panel submit | `"Filter"` (not "Search" — this is exact-match/multi-select narrowing of a known vocabulary, not free-text search) |
| Filter panel reset | `"Clear filters"` |
| `open_to_collaboration` filter "unset" option label | `"Any"` (relabel django-filter's default "Unknown" — "Any" reads correctly as D-07's unfiltered default state) |
| Empty state — campaigns list, zero qualifying `TargetList`s | Heading: `"No campaigns yet."` Body: `"A campaign appears here once at least one observing run has been logged against a target list. Import a coordination sheet or ask a coordinator to record the first run."` |
| Empty state — per-campaign table, filters produce zero rows | `Table.Meta.empty_text`: `"No runs match these filters. Clear filters to see all runs for this campaign."` with "Clear filters" as an inline link to the bare table URL |
| Error state — campaign `pk` does not exist | Standard Django 404 (`get_object_or_404` on `TargetList`); no custom "not found" template needed for this phase |
| Destructive confirmation | None in this phase — VIEW-01..04 is entirely read-only; no create/update/delete UI exists here. (Reserved for Phase 16's approval-queue actions.) |
