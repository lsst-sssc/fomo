<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client GET (target_pk / site_pk / end_date) → view | The dropdown scopes only what is *offered*; any GET param can be hand-crafted. The view must re-derive the allowed target/site sets and clamp the range server-side before use. |
| view → campaign_gap query + cache | Only validated, in-scope campaign/target/site/range values may reach the ORM query or the cache key. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-17-01 | Tampering / Elevation of Privilege | CampaignGapAnalysisView target_pk/site_pk (IDOR, cross-campaign leakage) | high | mitigate | Re-derive allowed target set (campaign.targets) and site set (campaign's used Observatories) server-side; validate submitted pk membership before any query or cache-key use; HttpResponseBadRequest() on mismatch (Pitfall 3, mirrors CampaignRunDecisionView). |
| T-17-02 | Denial of Service | CampaignGapAnalysisView date range | high | mitigate | View passes the client end date through clamp_date_range (Plan 01) — 180-day hard cap enforced server-side regardless of input, bounding per-request sun_event() cost. |
| T-17-03 | Information Disclosure | cache read/write in the view | medium | mitigate | View uses build_gap_cache_key with all four validated dimensions; a single-target's implied target is encoded explicitly, never omitted. |
| T-17-SC | Tampering | pip installs | low | accept | No new packages introduced by this plan (RESEARCH.md Package Legitimacy Audit: none). |
</threat_model>
