# Phase 16 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope:** SUBMIT-01..05, CAL-01..03 — a public submission form (`CampaignRunSubmissionView`), a
generic post-submission confirmation page, a staff-only approval-queue page (`ApprovalQueueView`,
D-01/D-02) with Approve/Reject actions, and the D-09 visibility change to Phase 15's existing
per-campaign table. Server-rendered Django templates (TOM Toolkit / Bootstrap 4), **not** a JS/SPA
component framework — same as Phase 15. The shadcn gate does not apply.

**Inherits from `15-UI-SPEC.md`:** this phase introduces **no new spacing scale, typography
scale, or base color palette** — every token below is either a direct reuse or a documented,
narrow extension of Phase 15's contract (e.g. reusing the already-reserved success green for a new
element). Read 15-UI-SPEC.md first; only genuinely new contract surface is spelled out in full
here.

## Spacing Scale

Unchanged from Phase 15 — reuse Bootstrap 4's `$spacer`-based utility classes (`p-*`, `m-*`,
`mb-*`) mapped to the 8-point scale. No new custom CSS for layout spacing.

| Token | Value | Bootstrap 4 utility | Usage |
|-------|-------|----------------------|-------|
| xs | 4px | `.p-1` / `.m-1` | Icon-to-text gaps (pending-count banner icon, action-button icon if used) |
| sm | 8px | `.p-2` / `.m-2` | Gap between Approve/Reject buttons in the actions column |
| md | 16px | `.p-3` / `.m-3` | Default `form-group` field spacing (crispy-forms default); card/panel padding |
| lg | 24px | `.p-4` / `.mb-4` | Page heading → form/table; Fieldset → Fieldset gap on the submission form |
| xl | 32px | not used this phase | — |
| 2xl | 48px | `.p-5` | Submission-thanks page vertical centering (matches Phase 15's empty-state `.p-5` pattern) |
| 3xl | 64px | not used this phase | — |

**Exceptions:** none new. The honeypot field (`alt_contact_info`) is rendered via
`widget=forms.HiddenInput()` — it occupies **zero** visual space and is exempt from the spacing
scale entirely (it must never be a spacing/layout consideration; see Honeypot Field Contract
below).

## Typography

Unchanged from Phase 15 — no new sizes/weights. Applies to all new pages this phase adds.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body / form field text / table cell text | 16px (`1rem`) | 400 | 1.5 |
| Small / metadata (help text, "pending N days" style hints if used) | 14px | 400 | 1.5 |
| Label / form field labels / table header / badge text | 14px | 700 (`.font-weight-bold`, explicit) | 1.2 |
| Page heading (e.g. "Submit an Observing Run", "Approval Queue") | 24px (`<h4>`, `.font-weight-bold`) | 700 | 1.2 |

**Crispy-forms note:** `django-crispy-forms`' bootstrap4 template pack renders field labels via
`<label>` at the browser default weight unless overridden — confirm the form's `FormHelper` does
not need an override; Bootstrap 4's default `<label>` is already effectively regular-weight body
text (14px is achieved via `.form-group small.form-text` for help text, which crispy applies
automatically). Do not introduce a third weight for form labels.

## Color

Unchanged base palette from Phase 15 (`custom.css` `:root`). This phase adds one **documented,
narrow extension**: reusing the already-reserved `--success` green for a new element (the Approve
button), not introducing a new hex value.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#ffffff` | Page background — submission form, approval queue, thanks page |
| Secondary (30%) | `#f8f9fa` (`--light`) | Form `<div class="card bg-light">` wrapper (matches Phase 15's filter-panel card); approval-queue section headers background |
| Accent (10%) | `#007bff` (`--primary`) | Reserved for: submission-form Submit button (`btn-primary`), "Submit a Run" entry-point button/link, sortable-column headers on the approval-queue tables (django-tables2 default) |
| Success (new usage, same hex as Phase 15's `approved` badge) | `#28a745` (`--success`) | **Reserved for: the Approve action button only** (`btn-success`). Phase 15 already assigned this exact hex to the `approval_status=approved` badge — reusing it here preserves the existing semantic meaning ("this is the approve/approved color") rather than introducing a second, disconnected green |
| Destructive/Alert | `#dc3545` (`--danger`) | Reserved for: `approval_status=rejected` badge (unchanged, Phase 15) **and** the Reject action button (`btn-danger`) — same semantic reuse pattern as Success above. **Not** used for any `run_status` value (unchanged Phase 15 rule) |
| Warning | `#ffc107` (`--warning`) | Reserved for: the staff-only "N submissions pending review" entry-point banner (matches the existing `pending_review` badge color from Phase 15 — same semantic reuse pattern) |

**Accent reserved for:** submission-form submit action, "Submit a Run" entry links, and
sortable-column headers only — never a general-purpose button color for this phase's new pages.

## Layout Contract

### Page 3 — Submission form (SUBMIT-01, public, unauthenticated)

`CampaignRunSubmissionView` (`FormView`), crispy-forms layout following the established
`EphemerisForm` pattern (`solsys_code/forms.py`) — see 16-RESEARCH.md Pattern 1 for the field list.

- Page heading: `<h4 class="font-weight-bold mb-4">Submit an Observing Run</h4>`
- Intro copy (one `<p>` above the form, matching `ephem_form.html`'s precedent of a short
  instructional line before the crispy render): see Copywriting Contract
- Form fields, in this visual order (matches D-05's field set and research Pattern 1's grouping):
  1. `campaign` (required `ModelChoiceField` dropdown) — rendered alone, above the fieldsets, so
     its "required" status is visually distinct from the optional fields below it
  2. Fieldset **"Run details"**: `telescope_instrument`, `site_raw` (label: "Observing site" per
     research Pattern 1 — not the model's verbose_name "Original site code text," which is
     staff/CSV-import jargon unsuitable for a public form), `obs_date`, `ut_start`, `ut_end`,
     `filters_bandpass`, `observation_details`, `open_to_collaboration`
  3. Fieldset **"Contact"**: `contact_person`, `contact_email`, `comments` — both contact fields
     marked required (red asterisk, Bootstrap 4/crispy default `asteriskField` convention),
     consistent with D-06
  4. Honeypot field (`alt_contact_info`) — see Honeypot Field Contract below; not part of any
     visible fieldset, does not appear in this ordering to a sighted user at all
  5. `FormActions`: `Submit('submit', 'Submit run for review')` (`btn-primary`, accent color) +
     a `Cancel` link back to the campaigns list (`btn-outline-primary`, matching
     `EphemerisForm`'s existing Cancel-link precedent) — see Copywriting Contract
- Field-level validation errors render via crispy-forms' default Bootstrap 4 `is-invalid` /
  `invalid-feedback` styling — no custom error template needed (matches project-wide Django form
  convention; no override exists anywhere else in this codebase either)
- Form-level error (Pitfall 4 — duplicate telescope+campaign+`ut_start` collision): render as a
  crispy `non_field_errors` banner (Bootstrap 4 default: `<div class="alert alert-danger">` at the
  top of the form) — see Copywriting Contract for exact text

### Page 4 — Submission-thanks page (shown for both genuine and honeypot-tripped submissions)

Plain `TemplateView`, no form, no table — reuses Phase 15's empty-state visual pattern
(`<div class="p-5 text-center text-muted">`) for a calm, low-emphasis confirmation:

```html
<div class="p-5 text-center">
  <h5 class="font-weight-bold">Thanks — your submission is pending review.</h5>
  <p class="text-muted">
    A staff member will review it shortly. Approved runs appear on the campaign table and,
    if a telescope and date range were provided, on the shared calendar.
  </p>
  <a href="{% url 'campaigns:list' %}" class="btn btn-outline-primary">Back to Campaigns</a>
</div>
```

**Critical:** this exact markup renders identically whether the submission was genuine or
honeypot-tripped (SUBMIT-04) — there must be no conditional branch in this template keyed on
honeypot state, and no query-string/status flag that would let a bot distinguish the two paths by
inspecting the response.

### Page 5 — Approval queue (D-01/D-02, staff-only)

`ApprovalQueueView` (`StaffRequiredMixin` + `TemplateView`). Structure top to bottom:

1. **Page heading:** `<h4 class="font-weight-bold mb-4">Approval Queue</h4>`
2. **Pending section** — header `<h5 class="font-weight-bold mb-3">Pending Review</h5>`, then
   `{% render_table pending_table %}` using `ApprovalQueueTable` (extends `CampaignRunTable`,
   Phase 15's 13-column staff view, plus one appended **Actions** column):
   - Actions column, one `<td>` per row containing two independent `<form method="post">`
     mini-forms side by side (`d-flex`, `.5rem`/8px gap between them — spacing token `sm`):
     - `<button type="submit" class="btn btn-sm btn-success">Approve</button>` — POSTs
       `action=approve` to `campaigns:decide` (or equivalent), **no confirmation dialog** (see
       Destructive Confirmation Contract below for why)
     - `<button type="submit" class="btn btn-sm btn-danger">Reject</button>` — POSTs
       `action=reject`, **requires a native `confirm()` dialog** before submitting (see below)
   - Empty state (zero pending rows): `Table.Meta.empty_text` — see Copywriting Contract
3. **Recently Decided section** — header `<h5 class="font-weight-bold mb-3 mt-4">Recently
   Decided</h5>`, then `{% render_table decided_table %}` — same `ApprovalQueueTable` class with
   `show_actions=False` (no Approve/Reject buttons; this section is read-only/informational per
   16-RESEARCH.md Open Question 2), capped at 20 rows, ordered `-pk`
   - Empty state (zero decided rows yet): see Copywriting Contract
4. Both tables use `django.contrib.messages` feedback (Bootstrap 4 alert, already wired) after a
   POST redirect back to this same page — success (`messages.success`, green `alert-success`) or
   the "already decided by someone else" race-condition case (`messages.warning`, yellow
   `alert-warning`) per 16-RESEARCH.md Pattern 4

### Entry points (discoverability — no locked decision on exact mechanism; this contract fixes it)

**"Submit a Run" entry point (SUBMIT-01 discoverability):**
- Add a `<a class="btn btn-primary" href="{% url 'campaigns:submit' %}">Submit a Run</a>` button
  to the **campaigns list page** (Phase 15 Page 1), placed to the right of the `<h4>` page heading
  in a `d-flex justify-content-between align-items-center` header row (mirrors Bootstrap 4's
  standard page-header-with-action-button idiom; does not disturb Phase 15's existing
  `list-group` body below it)
- Also add the same button to the **per-campaign table page** (Phase 15 Page 2), same placement
  relative to its `<h4>` heading, so a visitor already looking at one campaign's runs can submit
  directly without navigating back to the list first

**Approval-queue entry point (D-01 — "reachable from the existing Campaigns navbar entry"):**
- Do **not** turn the "Campaigns" navbar item into a dropdown — keep the single nav-item precedent
  from Phase 15 unchanged (no new visual style on the navbar itself, per Phase 15's own contract)
- Instead, add a **staff-only banner** at the top of the campaigns list page (reachable via the
  unchanged "Campaigns" navbar entry → list page → banner), shown only when
  `request.user.is_staff` and at least one `CampaignRun` has `approval_status='pending_review'`:
  ```html
  {% if request.user.is_staff and pending_count %}
  <div class="alert alert-warning d-flex justify-content-between align-items-center mb-4">
    <span><i class="fa fa-exclamation-circle mr-2"></i>{{ pending_count }} submission{{ pending_count|pluralize }} pending review</span>
    <a href="{% url 'campaigns:approval_queue' %}" class="btn btn-sm btn-warning">Review queue</a>
  </div>
  {% endif %}
  ```
  This reuses `--warning` (`#ffc107`), consistent with the `pending_review` badge's existing color
  meaning (Phase 15), and satisfies D-01 without adding a second navbar item or a dropdown.

## Honeypot Field Contract (SUBMIT-04)

- Rendered via `widget=forms.HiddenInput()` — never a CSS-only `display:none` on a normally-typed
  field. This is a hard requirement per 16-RESEARCH.md Pitfall 5, not a stylistic choice: a
  `HiddenInput` never appears in the rendered HTML as a fillable, visible control at all, whereas a
  CSS-hidden visible-type field is still discoverable by inspecting computed styles.
- Field name must **not** be a literal `honeypot`/`spam_trap`/similar — use a plausible-sounding
  name (research recommends `alt_contact_info`); this is a naming/security concern, not a visual
  one, and has no bearing on any token in this contract.
- No label, no help text, no `crispy_forms.layout.Field(...)` entry rendered in any fieldset — the
  field must not occupy a position in the visible Layout at all (belt-and-suspenders: if it must
  be referenced in `Layout(...)` for crispy to include it in the POST at all, wrap it in
  `Div('alt_contact_info', css_class='d-none')` as an extra layer of defense, per research
  Pattern 1 — but the `HiddenInput` widget alone is what does the real work).
- On trip: identical response to a genuine submission (Page 4, the thanks page) — see that
  section's "Critical" note. No error styling, no distinguishable markup, ever.

## Destructive Confirmation Contract

| Action | Confirmation? | Rationale |
|--------|---------------|-----------|
| **Approve** | None — single click, immediate POST | Approving is this queue's primary/expected action (the "happy path"); it is also reversible in spirit — this phase's data model has no hard delete, and a mis-approval is a staff-to-staff conversation, not data loss. Adding friction here works against the queue's core purpose (fast triage) |
| **Reject** | Native `confirm()` dialog before the POST fires: `onclick="return confirm('Reject this submission? The submitter will not be automatically notified.')"` on the Reject `<button>` | Rejecting is a one-way state transition in this phase (16-RESEARCH.md Open Question 2: no undo/re-open action is in scope) and the submitter gets no automatic feedback (SUBMIT-07, submitter status-check, is v2/out of scope) — the confirmation copy surfaces that real consequence rather than being generic "are you sure?" filler |
| **Double-approve / double-reject (race condition)** | No user-facing confirmation needed — handled silently and correctly by the atomic conditional update (SUBMIT-03); the second actor sees a `messages.warning` "This run was already decided by someone else" banner after their POST, not a confirmation dialog before it |

No JS framework/modal library is introduced for this — a native `confirm()` call is sufficient and
consistent with this codebase's zero-client-framework footprint (no Bootstrap 4 JS modal, no new
`<script>` bundle).
