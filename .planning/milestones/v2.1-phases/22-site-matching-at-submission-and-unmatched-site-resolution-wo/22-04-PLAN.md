<objective>
Close UAT gaps 1 and 3 (a single shared root cause): the live-search suggestion widgets
never render suggestions in a real browser even though the debounced `hx-get` fires and
returns HTTP 200. `SiteSearchView.get()` reads the search term exclusively via
`request.GET.get('q', '')`, but htmx's `hx-get` serializes only the triggering input's own
`name`-keyed value plus `hx-vals` (it does NOT gather enclosing-form fields for GET, unlike
POST). So the public form's `<input name="site_raw">` sends `?site_raw=<text>&input_id=…`
and the approval-queue widgets' `<input name="site_selection">` send
`?site_selection=<text>&input_id=…` — never `?q=<text>`. The view therefore always sees an
empty term, trips its `< 2` empty-fragment gate, and renders nothing on every keystroke.

Purpose: restore the D-09 (public form) and D-10 (approval-queue) live-search behavior that
markup-only Django TestCase assertions could not catch (they never execute JS/htmx). A single
view-side change fixes all three widget instances at once, with zero widget/template edits — so
every existing markup, escapejs, htmx-trigger-grammar, and throttle test stays green.

Output: `SiteSearchView.get()` resolves the term from `q` → `site_raw` → `site_selection`;
new regression tests in test_campaign_site_search.py that drive the view with the widgets'
actual param keys.
</objective>
