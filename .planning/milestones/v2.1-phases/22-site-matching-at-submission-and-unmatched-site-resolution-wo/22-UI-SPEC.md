# Phase 22 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope:** Two pieces. (1) A live HTMX fuzzy-search widget replacing the public submission
form's bare `site_raw` text field, and replacing the approval queue's static ≤5-candidate
`<datalist>` with the same live widget on pending rows. (2) A new "Sites needing review" third
table on the approval-queue page listing approved runs with `site_needs_review=True`, each row
carrying the same live-search widget plus a single "Resolve" action. Server-rendered Django
templates (TOM Toolkit / Bootstrap 4), **not** a JS/SPA component framework — same as Phases
15/16/21. The shadcn gate does not apply. `django-htmx` is already installed and its
`htmx.min.js` is already loaded site-wide (`tom_common/base.html`) — this phase is the first to
actually use it for a live client interaction, but introduces no new package.

**Inherits from `15-UI-SPEC.md`, `16-UI-SPEC.md`, and `21-UI-SPEC.md`:** this phase introduces
**no new spacing scale, typography scale, or base color palette** — every token below is a direct
reuse of the established contract. Read all three first; only genuinely new contract surface (the
live-search fragment, the sites-needing-review table, the Resolve action) is spelled out in full
here. Phase 22 supersedes Phase 21's *mechanism* for the Site column (static datalist → live HTMX
fragment) but changes no color, spacing, or typography token Phase 21 established.

## Color

Unchanged base palette from Phases 15/16/21 (`custom.css` `:root`). This phase adds **zero new
hex values** — every new element below reuses an already-reserved role.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#ffffff` | Page background — submission form, approval queue (unchanged) |
| Secondary (30%) | `#f8f9fa` (`--light`) | Suggestion-list item hover state (Bootstrap 4 `list-group-item-action` default hover background — no custom CSS needed, this *is* the framework default for that exact hex) |
| Accent (10%) | `#007bff` (`--primary`) | Reserved for: the **Resolve** action button (`btn-primary`, new this phase — see rationale below) and the unchanged "Create new Observatory" link. No other new use. |
| Success | `#28a745` (`--success`) | Approve button only (unchanged from Phase 16/21) — **not** reused for Resolve (see rationale below) |
| Destructive/Alert | `#dc3545` (`--danger`) | Reject button only (unchanged) — **not** reused for "no matches" or resolve-failure states, which are neutral/informational, not errors in the destructive sense (see Copywriting Contract) |
| Warning | `#ffc107` (`--warning`) | Unchanged — `pending_review` badge / staff-only banner only. A resolve **failure** (couldn't match to any obscode) uses `messages.warning` (row stays visible, action retryable — not a hard error) |

**Accent reserved for (this phase's addition):** the **Resolve** button. Rationale: Resolve is
neither the queue's "happy path bulk action" (that's Approve, which owns `--success` per Phase
16's semantic reservation) nor a rejection. It is a single-purpose, one-off completion action on
an already-approved run, structurally closer to the submission form's Submit button (`--primary`,
Phase 16) than to Approve/Reject. Reusing `--success` here would blur the "this run just got
approved" signal Approve already owns exclusively.

**Deliberate non-change:** suggestion-list rendering, badge colors, and every existing
`render_run_status`/`render_approval_status` treatment are untouched by this phase.

## Layout Contract

### Live-search suggestion fragment (D-01..D-06, new — shared markup, two call sites)

One partial template (`campaigns/partials/site_search_results.html`), rendered identically for
both the public form and the approval-queue widget:

```html
{% if candidates %}
<ul class="list-group" id="site-suggestions-list">
  {% for display, obscode in candidates %}
  <li class="list-group-item list-group-item-action"
      style="cursor:pointer;"
      onclick="document.getElementById('{{ input_id }}').value = '{{ display|escapejs }} ({{ obscode|escapejs }})'; document.getElementById('site-suggestions-{{ input_id }}').innerHTML = '';">
    {{ display }} ({{ obscode }})
  </li>
  {% endfor %}
</ul>
{% elif query %}
<p class="text-muted small mb-0">{{ no_matches_copy }}</p>
{% endif %}
```

- Suggestion display text is **always** `"{Display Name} ({obscode})"` (D-05) — no truncation, no
  separate "obscode" column; this is the same single-string convention Phase 21 established for
  the datalist `<option>` text.
- Every interpolated value (`display`, `obscode`, submitter-typed `query`) goes through Django's
  default auto-escaping for the visible text node **and** the `|escapejs` filter for the value
  embedded inside the inline `onclick=` JS-string context — these are two distinct escaping
  contexts (see Security note below); never `mark_safe` or f-string/`%` interpolation.
- Clicking a suggestion fills the input's `.value` and clears the suggestions container in one
  inline handler — no round trip, no page reload (D-06: text-only fill, resolution deferred).
- Below 2 characters typed: no request fires at all (`hx-trigger` event-filter bracket syntax,
  not `hx-min-length` — see Copywriting/interaction constants below). Nothing renders in the
  suggestions container in this state — it stays visually empty, not an explicit "type more"
  hint (avoids a distracting flicker on every keystroke below the threshold).
- Fewer than 2 chars is the *only* silent-empty state; 2+ chars with zero results renders the
  "no matches" hint (copy differs by call site — see Copywriting Contract).

**Interaction constants (locked this phase, not left open):**

| Constant | Value |
|----------|-------|
| Minimum characters before searching | 2 |
| `hx-trigger` debounce delay | 300ms (`keyup changed delay:300ms[this.value.length >= 2]`) |
| Suggestion count cap | 8 |
| Throttle limit | 40 requests / 60s per IP (staff sessions exempt — see Pitfall 5 discretion in RESEARCH.md; no visible UI change either way, a throttled request simply fails to swap and the user's next keystroke retries) |

### Public submission form — `site_raw` field (D-01, D-04..D-06, D-09, changed)

Replaces the current bare `CharField` rendering. No "Create new Observatory" link (D-09 — public
submitters never get a site-creation path):

```python
site_raw = forms.CharField(
    max_length=255,
    required=False,
    label='Observing site',
    widget=forms.TextInput(attrs={
        'hx-get': reverse_lazy('campaigns:site_search'),
        'hx-trigger': "keyup changed delay:300ms[this.value.length >= 2]",
        'hx-target': '#site-suggestions-id_site_raw',
        'hx-swap': 'innerHTML',
        'hx-vals': '{"input_id": "id_site_raw"}',
        'autocomplete': 'off',
        'placeholder': 'MPC code or site name…',
        'class': 'form-control',
    }),
)
```

The crispy `Layout` gets `HTML('<div id="site-suggestions-id_site_raw" class="mt-2"></div>')`
immediately after `'site_raw'` inside the existing **"Run details"** `Fieldset` (16-UI-SPEC.md
field order, unchanged) — same crispy `HTML()` injection mechanism Phase 21 already established
for a sibling field. This field keeps its normal `form-group` field spacing; the suggestions
container is not a modal/overlay, it pushes subsequent fields down (standard document flow, no
`position: absolute`) — consistent with this codebase's zero-JS-overlay footprint.

### Approval-queue pending row — `render_site()` (D-04, D-10, changed)

Replaces Phase 21's static ≤5-candidate `<datalist>` with the live fragment, **keeping** the
"Create new Observatory" link (D-10 — staff retain the escape hatch):

```html
<input type="text" name="site_selection" value="{site_raw}"
       id="site-input-{pk}" form="decide-form-{pk}"
       class="form-control form-control-sm" placeholder="MPC code or site name…"
       autocomplete="off"
       hx-get="{site_search_url}" hx-trigger="keyup changed delay:300ms[this.value.length >= 2]"
       hx-target="#site-suggestions-site-input-{pk}" hx-swap="innerHTML"
       hx-vals='{"input_id": "site-input-{pk}"}'>
<div id="site-suggestions-site-input-{pk}" class="mt-2"></div>
<a href="{create_observatory_url}" class="small ml-1">Create new Observatory</a>
```

- Visually identical row layout to Phase 21 (input, then link) with the datalist swapped for a
  live suggestions container placed directly below the input — same `.mt-2`/8px gap token used on
  the form.
- The `value="{site_raw}"` pre-fill and `format_html`/`format_html_join` escaping discipline
  carry over unchanged from Phase 21.
- Resolved rows and the read-only decided table keep Phase 21's/Phase 15's existing plain-text
  `render_site()` fallback — this phase touches only the actionable, unresolved-and-pending case.

### "Sites Needing Review" — new third table (D-07, D-08, new)

Placed on the existing approval-queue page, **after** the "Recently Decided" section, same page
(no new URL, no navbar entry — D-07):

```html
<h5 class="font-weight-bold mb-3 mt-4">Sites Needing Review</h5>
{% render_table review_table %}
```

- Reuses `ApprovalQueueTable` (D-07 — no new table class), instantiated against
  `CampaignRun.objects.filter(approval_status=APPROVED, site_needs_review=True)`, ordered `-pk`
  (mirrors the decided table's convention per CONTEXT.md's discretion note).
- **No row cap** — unlike the read-only "Recently Decided" audit log (capped at 20), this table is
  a live work queue of items genuinely needing staff action; capping it would hide actionable
  rows, which contradicts the phase's whole purpose of closing the "dead end" gap.
- Site column: same live-search widget as the pending row above, **plus** the same "Create new
  Observatory" link (Claude's discretion, not explicitly locked in CONTEXT.md — extended here for
  parity with SITE-02's general principle: "staff can resolve to an existing Observatory or
  explicitly create a new one," which this row is exactly the case for).
- Actions column: **single button, "Resolve"** (`btn-primary btn-sm`, accent color — see Color
  section rationale), submitting `action=resolve_site` via the same single-`<form>`-with-`form=`-
  attribute pattern Phase 21 established for Approve/Reject. No second button, no confirmation
  dialog (see Destructive Confirmation Contract).
- Empty state (zero rows needing review): see Copywriting Contract.
- Failure UX (D-08's resolve-failure case): the row **stays in the table**, unchanged, with a
  `messages.error` banner shown after the redirect — no inline per-row error styling is
  introduced (matches this page's existing messages-banner pattern for the race-condition case in
  16-UI-SPEC.md, not a new mechanism).

## Destructive Confirmation Contract

Unchanged from `16-UI-SPEC.md`/`21-UI-SPEC.md` for Approve/Reject. One new action this phase:

| Action | Confirmation? | Rationale |
|--------|---------------|-----------|
| **Resolve** (new, sites-needing-review table) | None — single click, immediate POST, same as Approve | Resolving is this table's only, intended, positive action — the run is already `APPROVED`; Resolve only fills in the site and (if eligible) projects the deferred calendar event. It never changes `approval_status`, never rejects, never deletes anything. Adding a confirmation dialog here would treat a corrective/completion action as if it were risky, which it structurally is not (matches Approve's "no friction on the happy path" rationale from 16-UI-SPEC.md) |
| **Resolve, failure path** | No dialog either — failure is reported *after* the attempt via a `messages.error`/`messages.warning` banner (see Copywriting Contract), not guarded against beforehand. The row stays retryable with no destructive side effect from a failed attempt. |

Picking a live-search suggestion (on any of the three widgets) is a text-fill only (D-06) — not a
submission, not destructive, no confirmation needed anywhere in this phase.
