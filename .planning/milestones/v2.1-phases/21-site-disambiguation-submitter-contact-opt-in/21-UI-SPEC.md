# Phase 21 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope:** SITE-01..03, VIEW-05 — a fuzzy-match site-disambiguation control inside the *existing*
approval-queue table's Site column (D-04), a refactor of that same row's Approve/Reject actions
from two independent `<form>`s into one (required so the new site input can target it via the
HTML5 `form=` attribute), a small extension to the existing `CreateObservatory` page (D-05/Pitfall
6: `?obscode=`/`?next=` prefill+redirect support), and one new checkbox on the public submission
form (D-07/VIEW-05). Server-rendered Django templates (TOM Toolkit / Bootstrap 4), **not** a
JS/SPA component framework — same as Phases 15-16. The shadcn gate does not apply.

**Inherits from `15-UI-SPEC.md` and `16-UI-SPEC.md`:** this phase introduces **no new spacing
scale, typography scale, or base color palette** — every token below is a direct reuse of the
established contract. Read both first; only genuinely new contract surface (the site-input
control, the actions-form refactor, the opt-in checkbox) is spelled out in full here.

## Layout Contract

### Approval-queue Site column — unresolved pending row (SITE-01/02, new)

Replaces the current plain muted-italic-text fallback **only** for rows where
`self.show_actions` is true (the pending table) **and** the row is unresolved (`run.site is
None`). Resolved rows and every decided-table row are unchanged (see Color section above).

```html
<input type="text" name="site_selection" value="{site_raw}" list="site-candidates-{pk}"
       form="decide-form-{pk}" class="form-control form-control-sm"
       placeholder="MPC code or site name…">
<datalist id="site-candidates-{pk}">
  <!-- 0-5 <option value="..."> entries, difflib.get_close_matches(n=5, cutoff=0.6) -->
</datalist>
<a href="{create_observatory_url}?obscode={site_raw|urlencode}&amp;next={approval_queue_url}"
   class="small ml-1">Create new Observatory</a>
```

- The `<input>` is **always rendered** for an unresolved pending row, whether or not any fuzzy
  candidates exist — an empty `<datalist>` degrades gracefully to a plain text input (per
  21-RESEARCH.md Pitfall 2: this input is the primary path for nickname-style site text like
  `'DCT'`, not a rare fallback shown only when the dropdown has candidates).
- The "Create new Observatory" link is **always visible next to the input**, never hidden behind
  a toggle, collapse, or shown-only-when-empty conditional — same rationale (Pitfall 2).
- `value="{site_raw}"` pre-fills the input with the submitter's original text so staff aren't
  starting from a blank field — matches `render_site()`'s existing "always show `site_raw`"
  discipline.
- All interpolated values (`site_raw`, candidate strings sourced from the external MPC API,
  `pk`) go through `format_html`/`format_html_join` with positional arguments — **never**
  `mark_safe` or f-string/`%` interpolation of submitter- or external-API-controlled text (matches
  `render_window_start()`'s existing tooltip precedent, stored-XSS defense per 21-RESEARCH.md
  Security Domain).

### Approval-queue Actions column — refactored single form (D-04, changed)

Currently renders **two independent** `<form>` elements (Approve, Reject) side by side. This
phase collapses them into **one** `<form id="decide-form-{pk}">` with two named submit buttons,
so the Site column's `<input form="decide-form-{pk}">` (a different `<td>` in the same `<tr>`)
can submit into it via the HTML5 `form=` attribute:

```html
<form id="decide-form-{pk}" method="post" action="{decide_url}">
  <input type="hidden" name="csrfmiddlewaretoken" value="{csrf_token}">
  <div class="d-flex" style="gap: 0.5rem;">
    <button type="submit" name="action" value="approve" class="btn btn-sm btn-success">Approve</button>
    <button type="submit" name="action" value="reject" class="btn btn-sm btn-danger"
            onclick="return confirm('Reject this submission? The submitter will not be automatically notified.')">Reject</button>
  </div>
</form>
```

- Visually **identical** to the current rendering (same button classes, same gap, same confirm
  dialog) — this is a structural/markup refactor for the `form=` wiring to work, not a visual
  change. The Destructive Confirmation Contract from 16-UI-SPEC.md carries over unchanged (Reject
  confirms, Approve does not).
- The single `<form>` still mints exactly one CSRF token, same as either of the two forms it
  replaces (16-RESEARCH.md/this phase's research: "does not remove or weaken CSRF protection").

### `CreateObservatory` page — `?obscode=`/`?next=` support (D-05/Pitfall 6, extended)

No new page — the existing `observatory_create.html` template and its intro copy are unchanged.
Two behavioral (non-visual) additions to the view:

- `?obscode=` in the query string pre-fills the form's obscode field as `initial=` — the rendered
  `{% bootstrap_form form %}` looks identical, just arrives pre-filled instead of blank.
- `?next=` (validated server-side) becomes the post-create redirect target instead of always
  going to the Observatory detail page. No new visible element — this is a redirect-destination
  change only, invisible in the page's own markup.

### Submission form — contact opt-in checkbox (D-07/VIEW-05, new)

One `forms.BooleanField(required=False)`, default unchecked, added to
`CampaignRunSubmissionForm`'s existing crispy `Layout`, placed **immediately after**
`contact_email` inside the existing **"Contact"** `Fieldset` (not a new fieldset — mirrors
`open_to_collaboration`'s existing placement style: a single un-boxed checkbox, no card/border,
rendered by crispy-forms' default Bootstrap 4 checkbox template):

```python
Fieldset('Contact', 'contact_person', 'contact_email', 'contact_public_opt_in', 'comments')
```

- Rendered as a standard Bootstrap 4 `custom-checkbox` (crispy-forms default), same visual
  weight/style as `open_to_collaboration` elsewhere on this same form — no custom CSS.
- Help text (see Copywriting Contract) renders via crispy's automatic `<small class="form-text
  text-muted">` — the existing "Label / metadata" typography role (14px/400), same mechanism
  already used for every other help-text-bearing field on this codebase's forms.
- No visual indicator of opt-in state is added to the approval queue (Claude's discretion, per
  21-RESEARCH.md Open Question 1 recommendation) — the approval queue stays a minimal,
  triage-focused column set (matches the existing UAT-14 gap-closure precedent that already
  excludes `weather`/`observation_outcome`/`publication_plans` from that table). The opt-in
  state is a per-campaign-table (post-approval) display concern only.

## Destructive Confirmation Contract

Unchanged from `16-UI-SPEC.md` — this phase adds no new destructive action. Approve/Reject retain
their existing confirmation rules exactly (Approve: none; Reject: native `confirm()`). Creating a
new `Observatory` via the extended `CreateObservatory` flow is additive/non-destructive (matches
its existing, un-confirmed "Create Observatory" submit button) and needs no new confirmation.
Checking the opt-in checkbox on the public submission form is a one-time, self-disclosed choice
by the submitter at submission time (D-07: no post-submission edit view exists this phase) — not
a destructive action, no confirmation needed.
