<objective>
Build the staff-facing inline site-disambiguation UI (SITE-01, D-04) inside the existing
approval-queue table: for an unresolved pending row, the Site column becomes an
`<input list=...>` + `<datalist>` of fuzzy-matched candidates plus an always-visible "Create new
Observatory" link, submitted into that row's approve/reject form via the HTML5 `form=` attribute —
no new endpoint, no AJAX, no JavaScript (RESEARCH Pattern 4). This requires collapsing the row's
two separate Approve/Reject `<form>`s into one so the site input can target it.

Purpose: Turn quick task `260705-l1v`'s site-visibility fix into an actual resolution control, using
the cached candidate pool + difflib matcher delivered by Plan 21-01.

Output: A single-`<form>` `render_actions()`, a `render_site()` override on `ApprovalQueueTable`,
a `candidate_pool` kwarg wired from `ApprovalQueueView.get_context_data`, and rendering + stored-XSS
tests. The decision-time behavior (D-06 clobber guard, reading `site_selection`) and the
`CreateObservatory` `?next=` handling land in Plan 21-04.
</objective>
