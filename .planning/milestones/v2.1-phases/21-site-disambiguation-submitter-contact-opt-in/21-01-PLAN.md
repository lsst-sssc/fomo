<objective>
Build the server-side backend for the approval-queue fuzzy-match site-disambiguation feature
(SITE-01, D-01/D-02/D-03): a bulk MPC obscode fetch, a cached merged candidate pool, and a
difflib-based fuzzy matcher — plus the Wave-0 `TestSiteFuzzyMatch` scaffold that the rest of the
phase's site work verifies against.

Purpose: Widen the fuzzy-match candidate pool from the ~8-row local `Observatory` table (Phase
18's spike found this too narrow) to the live MPC Obscodes list (~2,710 codes), cached idiomatically
so the approval-queue page never pays a 1.5 MB / ~1.3s live fetch per render (RESEARCH Pitfall 5).

Output: A `query_all()` bulk-fetch method, `build_site_candidates()` (cached) and
`fuzzy_match_candidates()` helpers, and a new test class with a reusable bulk-MPC fixture. No UI or
view wiring here — that lands in Plan 21-03, which consumes these helpers.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| External MPC API → backend | Untrusted external strings (`name_utf8`, `short_name`, `old_names`) enter the cached candidate pool |
| Submitter `site_raw` → fuzzy matcher | Submitter-controlled free text is fed to `difflib` and (later, in 21-03) rendered |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-21-01 | Tampering | MPC-sourced candidate strings entering the pool | medium | mitigate | Pool holds plain data only; all rendering escaping is enforced at display time in Plan 21-03 via `format_html`/`format_html_join`. No `mark_safe`/f-string interpolation of pool strings anywhere. |
| T-21-04 | Denial of Service | Per-request bulk MPC fetch (1.5 MB / ~1.3s) | high | mitigate | `build_site_candidates()` caches the merged pool (`django.core.cache`, 24h TTL) so the bulk fetch runs at most once per TTL, not per page render (Pitfall 5). |
| T-21-BF | Availability | MPC API outage during candidate build | medium | mitigate | Narrow `except` around the bulk fetch falls back to the local-only `Observatory` pool and never raises into the approval-queue render (RESEARCH Environment Availability). |
| T-21-SC | Tampering | npm/pip/cargo installs | n/a | accept | No packages installed this phase — `difflib` is stdlib; `django.core.cache`/`requests` are pre-existing (RESEARCH Package Legitimacy Audit: no packages to audit). No supply-chain checkpoint required. |
</threat_model>
