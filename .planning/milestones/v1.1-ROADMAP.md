# Milestone v1.1: Classical Run Ingest

**Status:** ✅ SHIPPED 2026-06-16

## Overview

Stage 2 of the "telescope runs on the calendar" feature (issue #37): a `load_telescope_runs`
management command that parses classical-schedule run lines and idempotently creates one
`tom_calendar.CalendarEvent` per observing night, using Stage 1's `sun_event()` for
accurate UTC sunset/sunrise times.

### Phase 3: Classical Calendar Ingest

**Goal**: Running `load_telescope_runs` against a file of classical run lines populates the calendar with one accurate, idempotent `CalendarEvent` per observing night for each parsed run.
**Depends on**: Phase 2 (Run Line Parsing); Phase 1 (Site & Ephemeris Helper)
**Requirements**: INGEST-01, INGEST-02, INGEST-03
**Plans**: 2 plans

**Wave 1**
- [x] 03-01-PLAN.md — Add `load_telescope_runs` management command (test-first) that expands parsed runs into idempotent nightly CalendarEvents (INGEST-01/02/03)

**Wave 2**
- [x] 03-02-PLAN.md — Phase 03 demo notebook (pre_executed) showing the ingest command and resulting CalendarEvents

**Details:**
`load_telescope_runs` Django `BaseCommand` reads a positional `filepath` argument. Per non-blank line: calls `parse_run_line()` → `get_site()` → `_iter_run_nights()` → `sun_event()` for each evening date. Upserts via `get_or_create` keyed on `(telescope, instrument, start_time)` with conditional `save()` (avoids `modified`-timestamp churn on unchanged re-run). Catches `(ValueError, Observatory.DoesNotExist)` per line (log+skip, not abort). End-of-run summary reports `created`/`updated`/`unchanged`/`skipped` counts. 6 `TestLoadTelescopeRuns` test methods cover all INGEST requirements and D-02/D-04 error paths; all 95 `./manage.py test solsys_code` tests pass.

Code review (03-REVIEW.md) found 4 issues: OSError handling for file open, TemporaryDirectory leak in notebook, assert guard for repo root, redundant return in `add_arguments` — all fixed in 03-REVIEW-FIX.md repair pass.

Microsecond-strip fix (commit `437aa53`): `astropy Time.to_datetime()` produces sub-second precision; `get_or_create` key stored timestamps with microseconds on first run, causing no-match on re-run (duplicate events). Fixed by rounding to second before the `get_or_create` call.

Demo notebook `docs/notebooks/pre_executed/load_telescope_runs_demo.ipynb` seeds 4 Observatory records, writes a sample 3-line schedule, invokes `load_telescope_runs` via `call_command`, and displays 8 `CalendarEvent` rows with `title`, UTC `start_time`/`end_time` (no microseconds), and 3-line `description`. Idempotency re-run shows `unchanged: 8`. Confirmed: `jupyter nbconvert --to notebook --execute` exits 0.

## Milestone Summary

**Key Decisions:**

- Telescope token resolved by prefix match against `SITES.keys()`; exact match wins, 2+ matches raise `ValueError` listing all candidates rather than guessing (Phase 2, D-01)
- Three date-range regex patterns tried in order: month-after-range, cross-month, month-before-range (Phase 2)
- `KNOWN_STATUSES` set matched case-insensitively; unmatched leftover word raises `ValueError` (Phase 2, D-06)
- `Observatory.DoesNotExist` caught alongside `ValueError` in per-line handler — both are log+skip, not abort (Phase 3, D-02)
- Cross-month runs (`day2 < day1`) raise `ValueError` in `_iter_run_nights` — fail loudly, not silently (Phase 3)
- CalendarEvent upsert keyed on `(telescope, instrument, start_time)` via `get_or_create` + conditional `save` — avoids duplicate events and `modified`-timestamp churn (Phase 3, D-04)
- CalendarEvent `description` format: dark-window ISO datetimes first, then `Status:`, then `Source line:` (Phase 3, D-06)
- `astropy Time.to_datetime()` microsecond-strip: round to second before DB save to ensure `get_or_create` key matches on re-run

**Issues Resolved:**

- v1.0 environment blocker (`tomtoolkit 3.0a9`/`tom_catalogs` mismatch) resolved before v1.1 start (main branch PR #38 merged 2026-06-11); `./manage.py test solsys_code` runs 95/95 OK
- `astropy Time.to_datetime()` microsecond accumulation in `get_or_create` key — produced duplicate events on re-run; fixed with `.replace(microsecond=0)`

**Issues Deferred:**
