# Phase 260723-02e Plan 01: Add --campaign flag to load_telescope_runs Summary

**Optional `--campaign NAME` flag on `load_telescope_runs` resolves a `tom_targets.TargetList` once upfront (fail-fast, explicit-name-only) and tags every created/updated `CalendarEvent.target_list` with it, matching the existing `backfill_lco_observation_records` and `sync_lco_observation_calendar` precedent.**

## Accomplishments
- `load_telescope_runs` accepts `--campaign NAME`, resolved once upfront via a new `Command._resolve_campaign()` before any schedule line is processed (fail fast on a bad name)
- `'target_list': campaign` is always included in the fields dict passed to `insert_or_create_calendar_event` for every observing night — `None` when the flag is omitted, the resolved `TargetList` when given
- 5 new tests cover: flag omitted (regression guard), matching campaign, no-match fail-fast, multiple-match fail-fast, and no-churn on re-run — all 23 tests in the file pass (18 pre-existing + 5 new)

## Decisions Made
- Duplicated the small explicit-name `TargetList` lookup locally in `load_telescope_runs.py` rather than factoring a shared helper with `backfill_lco_observation_records._resolve_campaign` — backfill's version also carries an interactive prompt-when-omitted branch that this command deliberately does not want, and the shared logic is only ~6 lines, so a shared abstraction would be over-engineering per the plan's explicit guidance.
- `'target_list'` is always present in the fields dict (never conditionally omitted), so the no-churn FK diff in `calendar_utils._update_or_unchanged` behaves consistently whether or not `--campaign` is passed on a given invocation.
