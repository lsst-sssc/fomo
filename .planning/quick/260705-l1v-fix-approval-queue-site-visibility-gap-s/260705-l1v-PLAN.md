<objective>
Fix two concrete defects in the campaign approval flow.

1. Visibility gap: the pending approval queue renders a blank Site column for every public
   submission, because `CampaignRunSubmissionView.form_valid()` deliberately leaves
   `site`/`site_needs_review` unset until approval (D-07), so `site_needs_review` is `False`
   (model default) while pending, and `CampaignRunTable.render_site()` only falls back to
   `site_raw` when that flag is `True`. Staff cannot see what site text was submitted before
   they approve/reject.

2. Silent bad data: `CampaignRunDecisionView.post()` calls `resolve_site(run.site_raw)`
   unconditionally on approve. For free-text facility nicknames (e.g. 'DCT', 'LCO-B') that
   match no existing Observatory (tier 1) and no MPC obscode (tier 2), tier 3 fabricates a
   placeholder `Observatory` row named `obscode='DCT'` -- polluting the Observatory table with
   a fake site. Public free-text submissions (unlike the already-vetted CSV import) should not
   auto-create these placeholders on approve.

Purpose: staff get visibility into submitted site text before deciding, and approving an
unresolvable free-text site no longer fabricates garbage Observatory rows -- while still
letting the run be approved (site resolution failure must not block approval, per D-07).

Output: a rendering fallback fix, a keyword-only opt-out on `resolve_site`, a one-line change
at the approval call site, and DB-backed tests proving both defects are fixed and neither the
CSV import path nor the spreadsheet-parity read table regresses.
</objective>
