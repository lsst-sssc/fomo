<objective>
Fix the calendar timezone-reset bug: the `#calendar-partial` div's `data-url` attribute in
`src/templates/tom_calendar/partials/calendar.html` is missing `utc_offset`, so the
`calRefresh`-triggered reload (fired after any event/todo create/update/delete in the vendored
`tom_calendar` package) always re-requests the calendar at UTC+0 and discards the user's chosen
timezone offset.

Purpose: a user who picks a non-zero UTC offset from the timezone `<select>` and then edits an
event should keep that offset after the automatic refresh, instead of the calendar snapping back
to UTC+0 (which re-buckets events across day boundaries and re-labels event times).

Output: a one-line template change adding `&utc_offset={{ utc_offset }}` to the `data-url`
query string (matching the existing Prev/Next/Today buttons), plus a regression test in the
existing `test_calendar_template.py` asserting the rendered `data-url` carries the selected
offset.

Scope note (CLAUDE.md paired-notebook rule): this change touches only a Django template and a
test file. It does NOT touch any of the four modules governed by the paired-demo-notebook rule
(`telescope_runs.py`, `load_telescope_runs.py`, `sync_lco_observation_calendar.py`,
`sync_gemini_observation_calendar.py`), so no notebook update is required.

Scope note (quality gates): `ruff` lints/formats Python only; the `.html` template is outside
its scope. The one Python file touched is the test module, which must keep `ruff check .` and
`ruff format --check .` clean. No view or other Python code changes — `utc_offset` is already in
the template context and already read by `fomo_render_calendar`.
</objective>

A non-zero value is deliberate: the buggy `data-url` omits `utc_offset` entirely, and a non-zero
offset proves the user's actual selection is threaded through (not just that a literal `0`
appears).

Recommended precise assertion (proves month/year/offset ordering is preserved):
resolve `url = reverse('calendar:calendar')`, then assert the response contains the exact
substring `data-url="{url}?month=6&year=2026&utc_offset=5"` via `self.assertContains`.
`assertContains` does a raw substring match and Django does not HTML-escape the literal `&`
characters in the template's data-url, so the literal `&`-joined string matches the rendered
output. If a future URL-prefix change makes the exact-match brittle, an acceptable fallback is
to regex-extract the `data-url="([^"]*)"` attribute value and assert `utc_offset=5` appears
inside it — but assert on the extracted data-url value specifically, not on a bare page-wide
occurrence of `utc_offset=5` (the Prev/Next/Today buttons also render it).

Keep the new test consistent with the file's existing style (plain `def test_...(self):`
methods, a short docstring, `self.client.get(reverse('calendar:calendar'), {...})`). No new
fixtures are needed — the setUp events are irrelevant to the data-url assertion. Do not modify
existing tests.
