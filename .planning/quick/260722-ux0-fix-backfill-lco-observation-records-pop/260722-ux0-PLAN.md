<objective>
Fix a real-world bug: backfilled LCO ObservationRecords always show a "[QUEUED]" prefix on
their calendar events even when status=COMPLETED, because `backfill_lco_observation_records`
creates records via `ObservationRecord.objects.create(...)` without ever setting
`scheduled_start`/`scheduled_end`. `sync_lco_observation_calendar._title_for()` decides the
[QUEUED] banner purely from `record.scheduled_start is None`, so backfilled terminal records
look perpetually queued.

Fix: after a record is created (non-dry-run, newly created only), call the facility's own
`update_observation_status(observation_id)` — the same TOM Toolkit method periodic polling uses
— which fetches live state and sets/saves `status`, `scheduled_start`, `scheduled_end`. Make it
best-effort: a failure is skip-and-logged and counted, never fatal, never rolls back the created
record.

Purpose: backfilled historical records get the same scheduled_start/scheduled_end TOM's normal
polling would populate, so the calendar reflects real LCO status.
Output: modified command + extended tests.
</objective>
