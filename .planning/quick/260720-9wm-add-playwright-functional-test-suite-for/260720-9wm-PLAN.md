<objective>
Add a Playwright-based functional test suite that verifies Bootstrap 5 rendering behavior
in the running FOMO app, as a follow-up to the tomtoolkit 3.0 / BS5 upgrade completed on
this branch (issue #45). Work stays on the current branch `issue45-tomtoolkit-3.0-upgrade`.

Purpose: the BS4->BS5 migration changed template tag libraries, crispy layout classes, and
the dropdown JS/data attributes. Unit-level template-tag checks cannot prove the BS5
JavaScript actually runs or that crispy emits BS5 layout markup. Real browser-driven
functional tests close that gap.

Output: `playwright` added to dev deps; a new Django `StaticLiveServerTestCase` suite of 3
browser tests under `solsys_code/tests/`; and a dedicated CI `functional-tests` job that
installs chromium and runs the suite with a cached SPICE kernel directory. Scope is exactly
these 3 files plus local verification — no calendar code, demo notebooks, or anything else.

Create a new Django test module using
`django.contrib.staticfiles.testing.StaticLiveServerTestCase` and
`playwright.sync_api.sync_playwright`. Give the module a Google-style module docstring and

D101/D102 but a short class docstring documents this new pattern, per CLAUDE.md conventions).

Browser lifecycle:
- `setUpClass`: call `super().setUpClass()`, then start `sync_playwright()` (store the manager
  so it can be stopped later) and launch a headless chromium browser, held as a class attribute.
- `tearDownClass`: close the browser, stop the playwright manager, then `super().tearDownClass()`.
- `setUp`: open a new browser page (store on the instance).
- `tearDown`: close the page.

Build absolute URLs from `self.live_server_url` plus the reverse() of each view name (import
`django.urls.reverse`). Do NOT hardcode paths.

Test 1 — navbar dropdown (proves BS5 JS runs, not just template presence):
- goto `self.live_server_url + '/'` (the site root; the Observatories dropdown is injected
  into the navbar there).
- The TOM navbar contains multiple `[data-bs-toggle="dropdown"]` toggles, so click the FIRST
  match (use Playwright locator `.first`) to avoid strict-mode "multiple elements" errors.
- After clicking, wait for a `.dropdown-menu.show` element to become visible (BS5 adds the
  `show` class when the JS toggles the menu). Assert it is visible. If `/` renders no
  toggle for anonymous users, re-verify against the running app and scope goto to a page that
  does show the Observatories nav item — but the site root is expected to work.

Test 2 — ephemeris form (proves the BS4->BS5 crispy migration):
- Create a fixture Target with `tom_targets.tests.factories.NonSiderealTargetFactory`
  (NEVER SiderealTargetFactory — FOMO is non-sidereal only, per CLAUDE.md). Do the creation
  inside the test (or setUp) so it lives in the test DB the live server serves.
- goto `reverse('makeephem', kwargs={'pk': target.pk})` joined to live_server_url.
- Assert the count of `.form-row` elements is exactly 0 (BS4 removed that class in BS5 crispy).
- Assert the count of `form .row` elements is >= 1 (the EphemerisForm crispy layout wraps its
  fields in Div(css_class='row'); confirmed present in solsys_code/forms.py).

Test 3 — observatory create form:
- goto `reverse('solsys_code_observatory:create')` joined to live_server_url.
- Fill the input named `obscode` (confirmed the literal field name) with a 3-character MPC
  site code, then submit the form (click the submit button) and wait for the resulting
  navigation to settle.
- Assert the current page URL contains `/observatory/`. This holds whether the live MPC
  lookup succeeds (redirect to /observatory/<pk>/) or fails (form re-renders at
  /observatory/create/) — the submit hits the real MPC API in CreateObservatory.form_valid();
  no mocking is required because the assertion is intentionally tolerant of both outcomes.

Follow the repo's snake_case + Google-style docstring conventions and single-quote formatting
so ruff stays clean.
