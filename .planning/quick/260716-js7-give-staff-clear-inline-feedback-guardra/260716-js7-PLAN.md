<objective>
Give staff a client-side nudge before they Approve a Pending Review row with an unresolved or
ambiguous Observing Site. When the paired site-search input is not in a "known resolved" state
(the staff member never clicked an actual dropdown suggestion), clicking Approve pops a
confirm() dialog first — mirroring the existing Reject-button confirmation pattern — so they
can notice and pick the site before it silently lands in Sites Needing Review. Cancelling the
dialog stops the submission; confirming proceeds exactly as today.

Purpose: Close the pure-UX gap identified in CONTEXT.md — the pending row's site_selection
input already submits alongside Approve (form="decide-form-{pk}") and the server already
resolves it, but nothing signals, before the click, whether the current field value will
actually resolve.

Output: A confirm-before-approve guard wired entirely in the table-render markup + the
suggestion fragment, plus Django test-client tests covering the mechanically verifiable
markup (guard/data-attribute presence for a pending row, absence for a decided/read-only row).
No backend, endpoint, or site-resolution-logic changes.
</objective>

  <files>solsys_code/campaign_tables.py, src/templates/campaigns/partials/site_search_results.html</files>
  <action>
Three coordinated markup edits, no backend or resolution-logic changes (do NOT touch
CampaignRunDecisionView.post(), resolve_site(), or selection_to_obscode()). The tracking key
is a `data-site-resolved` attribute on the site_selection input, read/written in JS as
`dataset.siteResolved` (camelCase mapping of the hyphenated attribute). Use the same value
convention everywhere: the string `'true'` means known-resolved, anything else means not.

1. In `solsys_code/campaign_tables.py`, `_render_site_search_widget()`: extend the
   `<input name="site_selection" ...>` element's attribute list (inside the existing
   format_html literal) with two additions — an initial `data-site-resolved="false"` state,
   and an `oninput` handler that resets it on manual typing: `oninput="this.dataset.siteResolved = 'false';"`.
   Both are STATIC literals in the format_html template (no `{n}` substitution, like the
   existing hx-trigger string), so they interpolate no untrusted data and need no escaping.
   Rationale: the suggestion-click fills the value programmatically via `inputEl.value = ...`,
   which does NOT dispatch an input event, so the click's known-resolved flag survives; only a
   real keystroke/paste (native input event) fires this oninput and clears the flag.

2. In `src/templates/campaigns/partials/site_search_results.html`: in the existing suggestion
   `onclick` handler, AFTER the line that sets `inputEl.value = displayText + ' (' + obscodeText + ')';`,
   add one static statement `inputEl.dataset.siteResolved = 'true';` (before or after the line
   that clears the suggestions container — order does not matter). This is a static literal that
   interpolates NO template variable, so it does not participate in the |escapejs contexts the
   file's head comment describes — leave every existing `{{ ...|escapejs }}` interpolation
   byte-for-byte unchanged; do not add mark_safe/f-string/% interpolation.

3. In `solsys_code/campaign_tables.py`, `render_actions()`, the pending-mode branch (the
   `format_html` that renders the Approve/Reject button pair): add an `onclick` guard to the
   Approve `<button ... value="approve" ...>` ONLY (leave the Reject button's existing confirm
   onclick unchanged; do NOT add a guard to the resolve-mode Resolve button — out of scope per
   CONTEXT.md). The guard, expressed in the same `onclick="..."` JS-string-in-HTML-attribute
   style the Reject button already uses (single quotes escaped as `\'` in the Python literal),
   must: look up the paired input via `document.getElementById('site-input-<pk>')`; if that
   element is missing, or its `value.trim()` is empty, or its `dataset.siteResolved === 'true'`,
   allow the submit (`return true`); otherwise `return confirm(...)` with the apostrophe-free
   message: This observing site does not look resolved yet. Approve anyway? It will land in
   Sites Needing Review. Interpolate the row pk as a new positional format_html argument
   (e.g. `{3}` -> `record.pk`) so the id string reads `site-input-{3}`; format_html escapes the
   int safely. Because the JS if-block uses literal `{ }` braces, double them (`{{ }}`) inside
   the format_html template string so they are not read as substitution placeholders. Keep the
   message free of apostrophes so it does not terminate the single-quoted JS string. The
   `input_id` produced by `render_site()` is `f'site-input-{pk}'`, so the guard's
   getElementById target matches it exactly; the null-element branch correctly no-ops the guard
   for an already-resolved pending row (plain-text render, no input) — that row approves with no
   dialog.
